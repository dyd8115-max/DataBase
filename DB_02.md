# SQL with PostgreSQL

---

## 사전 지식

### Docker / Docker Compose
- **Docker** : 애플리케이션을 컨테이너로 실행하는 플랫폼
- **Docker Compose** : 여러 컨테이너를 한 번에 설정하고 실행하는 도구 (`docker-compose.yml` 파일로 관리)
- **컨테이너** : 격리된 실행 환경. PostgreSQL 서버를 로컬에 직접 설치하지 않고 컨테이너로 띄울 수 있음

### psql
- PostgreSQL 전용 CLI 클라이언트. DB 서버에 접속해서 SQL을 직접 입력하는 도구
- `\` 로 시작하는 명령은 psql 내부 명령, `\!` 로 시작하면 쉘 명령

### 멱등성 (Idempotent)
- 같은 요청을 여러 번 반복해도 결과가 처음 한 번과 똑같은 성질
- 예: 결제 버튼을 두 번 눌러도 결제가 한 번만 되어야 함

## 1. PostgreSQL 환경 설정

### Docker로 PostgreSQL 서버 실행

```yaml
# docker-compose.yml
version: "3.9"
services:
  postgres:
    image: postgres:16
    container_name: pg-lab
    environment:
      POSTGRES_USER: dweb
      POSTGRES_PASSWORD: 1234
      POSTGRES_DB: dweb_db
    ports:
      - "5432:5432"
    volumes:
      - pgdata:/var/lib/postgresql/data
      - ./init:/docker-entrypoint-initdb.d
    restart: unless-stopped
volumes:
  pgdata:
```

```terminal
$ docker-compose up -d          # 컨테이너 백그라운드 실행
$ docker compose down --rmi all -v  # 컨테이너 + 이미지 + 볼륨 전체 삭제
```

### psql 클라이언트 접속

```terminal
# 컨테이너 내부에서 실행
$ docker exec -it pg-lab psql -U dweb -d dweb_db

# 로컬 머신에서 실행 (psql 설치 필요)
$ psql -h localhost -p 5432 -U dweb -d dweb_db
```

psql 설치:
```terminal
# Ubuntu
$ sudo apt update && sudo apt install postgresql-client

# macOS
$ brew install postgresql
```

### psql 주요 명령어

| 명령어 | 설명 |
|--------|------|
| `\l` | 모든 데이터베이스 목록 |
| `\dt` | 현재 DB의 테이블 목록 |
| `\d 테이블명` | 테이블 스키마(구조) 확인 |
| `\d+ 테이블명` | 테이블 상세 정보 확인 |
| `\c 데이터베이스명` | 다른 DB로 전환 |
| `\i 파일경로` | 파일에서 SQL 명령 실행 |
| `\q` | psql 종료 |
| `\! clear` | 터미널 화면 클리어 |
| `help` | 도움말 |

## 2. Database & Table 생성

### Database 생성 / 삭제

```sql
CREATE DATABASE db_lab;   -- 생성
DROP DATABASE db_lab;     -- 삭제
```

### Table 생성 / 삭제

```sql
-- 생성
CREATE TABLE student_lists (
    student_id      INT,
    student_name    VARCHAR(100),
    enrollment_date DATE

-- 삭제 (테이블 + 데이터 모두 삭제)
```

## 3. 데이터 삽입 — INSERT

```sql
-- 기본
INSERT INTO my_contacts (last_name, first_name, email)
VALUES ('Anderson', 'Jillian', 'jill@example.com');

-- 컬럼 순서 변경 가능
INSERT INTO file_logs (created_at, file_data, event_date)
VALUES ('2026-03-01 14:30:00', 'Hello Binary', '2005-11-04');

-- 컬럼 이름 생략 (테이블 컬럼 순서대로 입력)
INSERT INTO file_logs
VALUES ('Hello Binary', '2026-03-01 14:30:00', '2005-11-04');

-- 일부 컬럼만 입력 (나머지는 NULL 또는 DEFAULT)
INSERT INTO file_logs (file_data, event_date)
VALUES ('Hello Binary', '2005-11-04');
```

> 더미 데이터 생성 도구: [Mockaroo](https://www.mockaroo.com), [Generatedata](https://generatedata.com)

### INSERT ... SELECT

다른 테이블에서 조회한 결과를 그대로 INSERT:

```sql
INSERT INTO orders (email, product, qty)
SELECT email, 'Keyboard', 1
FROM source_users
WHERE email = 'alice@example.com';
```

## 4. 데이터 조회 — SELECT

### 기본 조회

```sql
SELECT * FROM my_contacts;                          -- 전체 컬럼
SELECT last_name, first_name FROM my_contacts;      -- 특정 컬럼만
```

### AS — 컬럼 별칭

실제 테이블 구조는 바뀌지 않고 조회 결과에서만 이름이 바뀜.

```sql
SELECT name AS username FROM users;
SELECT profession AS job FROM my_contacts;
SELECT 10*5 AS result;
```

실무 활용:
```sql
-- 프론트엔드가 원하는 JSON 키 이름으로 맞춰줄 때
SELECT user_id AS "userId", created_at AS "createdAt" FROM users;

-- JOIN 시 컬럼 이름 충돌 방지 (뒤에서 등장)
```

### WHERE — 조건 필터링

```sql
SELECT * FROM my_contacts WHERE status = 'married';
SELECT * FROM my_contacts WHERE profession = 'Artist' OR status = 'married';
SELECT * FROM my_contacts WHERE profession = 'Computer Programmer' AND status = 'committed';
SELECT * FROM my_contacts WHERE (status = 'married' OR status = 'divorced') AND location = 'San Francisco, CA';
```

### LIKE — 패턴 검색

| 와일드카드 | 의미 |
|-----------|------|
| `%` | 0개 이상의 임의 문자 |
| `_` | 정확히 1개의 임의 문자 |

```sql
SELECT * FROM my_contacts WHERE location LIKE '%CA';        -- CA로 끝나는 값
SELECT * FROM my_contacts WHERE email LIKE '_____@%';       -- @이전에 정확히 5글자
```

### BETWEEN — 범위 검색

```sql
SELECT * FROM my_contacts WHERE birthday BETWEEN '1975-11-1' AND '1980-12-31';
-- 아래와 동일
SELECT * FROM my_contacts WHERE birthday >= '1975-11-1' AND birthday <= '1980-12-31';
```

### IN — 다중 값 검색

OR 대신 IN으로 간결하게:

```sql
SELECT * FROM my_contacts
WHERE profession IN ('Student', 'Baker', 'Computer Programmer');

-- 위와 동일
SELECT * FROM my_contacts
WHERE profession = 'Student' OR profession = 'Baker' OR profession = 'Computer Programmer';
```

### 비교 연산자

| 연산자 | 의미 |
|--------|------|
| `=` | 같다 |
| `<>` | 같지 않다 (표준) |
| `!=` | 같지 않다 (비표준, 대부분 지원) |
| `>` / `>=` | 크다 / 크거나 같다 |
| `<` / `<=` | 작다 / 작거나 같다 |

## 5. 고급 SELECT

### ORDER BY — 정렬

```sql
SELECT * FROM my_contacts ORDER BY first_name;           -- 오름차순 (기본)
SELECT * FROM my_contacts ORDER BY first_name DESC;      -- 내림차순
SELECT * FROM my_contacts ORDER BY first_name, birthday; -- 여러 컬럼으로 정렬
```

### DISTINCT — 중복 제거

```sql
SELECT DISTINCT first_name FROM my_contacts;  -- 중복된 이름은 한 번만 출력
```

### 집계 함수

```sql
SELECT MIN(price) FROM store_sales;
SELECT MAX(price) FROM store_sales;
SELECT AVG(price) FROM store_sales;
SELECT SUM(quantity) FROM store_sales;
SELECT SUM(price * quantity) FROM store_sales;  -- 전체 매출 (price가 NULL이면 제외)
SELECT COUNT(price) FROM store_sales;           -- NULL 제외하고 카운트
SELECT COUNT(*) FROM store_sales;               -- 전체 행 카운트
```

### COALESCE — NULL 대체

```sql
-- price가 NULL이면 0으로 대체
SELECT product_name, COALESCE(price, 0) FROM store_sales;

-- NULL을 포함해서 평균 계산하고 싶을 때
SELECT AVG(COALESCE(price, 0)) FROM store_sales;
```

### NULLIF — 특정 값을 NULL로

```sql
SELECT NULLIF(5, 5);   -- NULL (같으면 NULL 반환)
SELECT NULLIF(5, 3);   -- 5   (다르면 첫 번째 값 반환)

-- 0으로 나누기 방지
SELECT price / NULLIF(quantity, 0) FROM store_sales;

-- COALESCE와 조합: price가 0이면 999로 처리
SELECT COALESCE(NULLIF(price, 0), 999) FROM store_sales;
```

### GROUP BY — 그룹 집계

```sql
SELECT status, COUNT(*) FROM my_contacts GROUP BY status;
SELECT main, COUNT(*), SUM(amount1) FROM easy_drinks GROUP BY main ORDER BY main;
SELECT category, AVG(price) FROM store_sales GROUP BY category;
```

### HAVING — 그룹 필터링

GROUP BY 이후 그룹 조건 필터링. WHERE는 행 단위, HAVING은 그룹 단위.

```sql
SELECT main, SUM(amount1)
FROM easy_drinks
GROUP BY main
HAVING SUM(amount1) >= 3.0;
-- 1. main으로 그룹화
-- 2. 그룹별 합계 계산
-- 3. 합계가 3.0 이상인 그룹만 반환
```

## 7. ON CONFLICT (충돌 처리)

### DO NOTHING — 중복 무시

```sql
INSERT INTO newsletter_users(email)
VALUES ('alice@example.com')
ON CONFLICT (email) DO NOTHING;
-- 이미 존재하면 에러 없이 무시
```

대량 삽입 시:
```sql
INSERT INTO newsletter_users(email)
SELECT email FROM staging_list
ON CONFLICT (email) DO NOTHING;
```

성공한 행만 반환:
```sql
INSERT INTO newsletter_users(email)
SELECT email FROM staging_list
ON CONFLICT (email) DO NOTHING
RETURNING email;
```

### DO UPDATE — UPSERT (없으면 INSERT, 있으면 UPDATE)

```sql
INSERT INTO products (product_id, product_name, stock)
VALUES (1, 'macbook', 5)
ON CONFLICT (product_id) DO UPDATE
SET stock = products.stock + EXCLUDED.stock;
-- product_id=1이 없으면 새로 INSERT
-- 있으면 기존 재고 + 새 입고량으로 UPDATE
-- EXCLUDED : INSERT하려고 했던 값
```

| 상황 | 문법 |
|------|------|
| 이미 있으면 무시 | `ON CONFLICT DO NOTHING` |
| 이미 있으면 갱신 | `ON CONFLICT DO UPDATE` |

## 8. 제약 조건 (Constraints)

### Primary Key (PK)

각 row를 유일하게 식별하는 컬럼. 중복·NULL 불가. 테이블당 1개.

```sql
CREATE TABLE orders (
    order_id BIGSERIAL PRIMARY KEY,  -- 자동 증가 정수 PK
    customer_id INT,
    order_date DATE,
    amount INT
);
```

> `BIGSERIAL` = 자동 증가하는 정수. `INT GENERATED ALWAYS AS IDENTITY`와 동일한 역할.

PK 나중에 추가:
```sql
ALTER TABLE orders ADD COLUMN order_id BIGSERIAL;
ALTER TABLE orders ADD PRIMARY KEY (order_id);
```

PK 제거:
```sql
ALTER TABLE orders DROP CONSTRAINT orders_pkey;
```

### UNIQUE

컬럼 값 중복 불가 제약. NULL은 허용. 테이블당 여러 개 가능.

```sql
ALTER TABLE students ADD CONSTRAINT unique_student_email UNIQUE (email);
ALTER TABLE students ADD UNIQUE (email);           -- 이름 생략 가능 (자동 생성)
ALTER TABLE students DROP CONSTRAINT unique_student_email;  -- 제거
```

| | PK | UNIQUE |
|--|:--:|:--:|
| 중복 금지 | ✅ | ✅ |
| NULL 허용 | ❌ | ✅ |
| 테이블당 개수 | 1개 | 여러 개 |
| 자동 인덱스 | ✅ | ✅ |
| 역할 | 행의 정체성 | 업무 규칙 |

### CHECK

비즈니스 로직을 DB 레벨에서 강제.

```sql
-- 나이 제한
CREATE TABLE users (
    user_id INT,
    age INT,
    CONSTRAINT check_age_range CHECK (age BETWEEN 0 AND 120)
);

-- 성별 값 제한
CONSTRAINT check_gender CHECK (gender IN ('M', 'F'))

-- 금액 음수 불가
CONSTRAINT check_amount_positive CHECK (amount >= 0)
```

나중에 추가:
```sql
ALTER TABLE exam_scores
ADD CONSTRAINT check_score_range CHECK (score BETWEEN 0 AND 100);
```

> 앱에서만 검사하면 버그나 우회로 뚫릴 수 있음. DB에서 검사하면 어떤 경로로 들어와도 규칙이 강제됨.

## 9. 다중 테이블 설계

### 단일 테이블의 문제점

모든 정보를 하나의 테이블에 넣으면 3가지 이상 현상 발생:

| 이상 현상 | 설명 |
|----------|------|
| **갱신 이상** (Update Anomaly) | 중복 저장된 데이터 일부만 수정하면 불일치 발생 |
| **삭제 이상** (Delete Anomaly) | 데이터 삭제 시 의도치 않게 다른 정보도 함께 삭제 |
| **삽입 이상** (Insert Anomaly) | 특정 정보만 독립적으로 삽입 불가 |

예: 사용자 정보 + 주문 정보가 한 테이블에 있으면, Alice의 마지막 주문을 삭제하면 Alice 고객 정보 자체가 사라짐.

### 정규화 (Normalization)

중복을 제거하고 데이터를 여러 테이블로 분리하는 과정.

### Foreign Key (FK)

한 테이블의 컬럼이 다른 테이블의 PK를 참조하는 제약 조건. 두 테이블 간의 관계를 설정하고 **참조 무결성**을 보장.

```sql
CREATE TABLE orders (
    order_id   SERIAL PRIMARY KEY,
    user_email TEXT NOT NULL REFERENCES users(user_email),  -- FK
    product    TEXT NOT NULL,
    qty        INT NOT NULL
);
```

- **참조 무결성** : FK가 가리키는 값이 반드시 부모 테이블에 존재해야 함
- FK가 없으면 → 주문은 있는데 고객이 없는 **고아 데이터(orphan row)** 발생
- 일대다 관계에서 FK는 항상 **N(다수) 쪽**에 위치

### 다중 테이블 INSERT

```sql
-- 고객 먼저 등록 (테이블 분리로 주문 없이 저장 가능)
INSERT INTO users(user_email, user_name, user_phone)
VALUES ('alice@inu.ac.kr', 'Alice', '010-1111-2222');

-- 주문 등록 (user_email을 조회해서 FK로 연결)
INSERT INTO orders(user_email, product, qty)
SELECT user_email, 'Keyboard', 1
FROM users
WHERE user_email = 'alice@inu.ac.kr';
```

### ON DELETE 옵션

부모 데이터 삭제 시 자식 데이터 처리 방법:

| 옵션 | 동작 |
|------|------|
| `RESTRICT` (기본값) | 자식이 참조 중이면 부모 삭제 막음 |
| `CASCADE` | 부모 삭제 시 자식도 함께 삭제 |

```sql
-- CASCADE로 변경
ALTER TABLE orders DROP CONSTRAINT orders_user_email_fkey;
ALTER TABLE orders ADD CONSTRAINT orders_user_email_fkey
    FOREIGN KEY (user_email) REFERENCES users(user_email)
    ON DELETE CASCADE;
```

## 10. JOIN

정규화로 분리된 테이블을 다시 연결해서 조회.

### INNER JOIN

양쪽 테이블에 모두 대응되는 행이 있을 때만 결과에 포함.

```sql
SELECT u.user_email, u.user_name, o.order_id, o.product, o.qty
FROM users u
INNER JOIN orders o ON u.user_email = o.user_email
ORDER BY o.order_id;
-- 주문이 없는 사용자는 결과에 포함되지 않음
```

### LEFT JOIN

왼쪽 테이블의 모든 행을 출력. 오른쪽에 없는 값은 NULL로 반환.

```sql
SELECT u.user_email, u.user_name, o.order_id, o.product
FROM users u
LEFT JOIN orders o ON u.user_email = o.user_email
ORDER BY o.order_id;
-- 주문이 없는 사용자도 포함 (주문 컬럼은 NULL)
```

### CROSS JOIN (주의)

JOIN 조건 없이 조회하면 두 테이블의 모든 행을 서로 다 조합함.

```sql
-- 아래 두 쿼리는 동일 (의도하지 않은 결과 주의)
SELECT * FROM users u, orders o;
SELECT * FROM users u CROSS JOIN orders o;
```

## 핵심 요약

```
DDL  : CREATE, DROP (구조)
DML  : INSERT, SELECT, UPDATE, DELETE (데이터)

SELECT 절 순서
  SELECT → FROM → WHERE → GROUP BY → HAVING → ORDER BY

제약 조건
  PK      : 행 식별, 중복·NULL 불가, 1개
  UNIQUE  : 중복 불가, NULL 허용, 여러 개
  CHECK   : 비즈니스 로직 강제
  FK      : 테이블 간 관계, 참조 무결성 보장

단일 테이블 문제
  갱신 이상 / 삭제 이상 / 삽입 이상 → 정규화로 해결

JOIN
  INNER JOIN : 양쪽 모두 있는 행만
  LEFT JOIN  : 왼쪽 테이블 전체 + 오른쪽 없으면 NULL
  CROSS JOIN : 모든 조합 (주의)

ON CONFLICT
  DO NOTHING : 중복 무시
  DO UPDATE  : UPSERT (없으면 INSERT, 있으면 UPDATE)
```
