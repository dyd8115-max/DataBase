# SQL with PostgreSQL

---

## 사전 지식

### Docker / Docker Compose
- **Docker** : 애플리케이션을 컨테이너로 실행하는 플랫폼
- **Docker Compose** : 여러 컨테이너를 한 번에 설정하고 실행하는 도구 (`docker-compose.yml` 파일로 관리)
- **컨테이너** : 격리된 실행 환경. PostgreSQL 서버를 로컬에 직접 설치하지 않고 컨테이너로 띄울 수 있음

### psql
- PostgreSQL 전용 CLI 클라이언트. DB 서버에 접속해서 SQL을 직접 입력하는 도구
- `\` 로 시작하는 명령은 psql 내부 명령, `\!` 로 시작하면 쉘 명령

### 멱등성 (Idempotent)
- 같은 요청을 여러 번 반복해도 결과가 처음 한 번과 똑같은 성질
- 예: 결제 버튼을 두 번 눌러도 결제가 한 번만 되어야 함

## 1. PostgreSQL 환경 설정

### Docker로 PostgreSQL 서버 실행

```yaml
# docker-compose.yml
version: "3.9"
services:
  postgres:
    image: postgres:16
    container_name: pg-lab
    environment:
      POSTGRES_USER: dweb
      POSTGRES_PASSWORD: 1234
      POSTGRES_DB: dweb_db
    ports:
      - "5432:5432"
    volumes:
      - pgdata:/var/lib/postgresql/data
      - ./init:/docker-entrypoint-initdb.d
    restart: unless-stopped
volumes:
  pgdata:
```

```terminal
$ docker-compose up -d          # 컨테이너 백그라운드 실행
$ docker compose down --rmi all -v  # 컨테이너 + 이미지 + 볼륨 전체 삭제
```

### psql 클라이언트 접속

```terminal
# 컨테이너 내부에서 실행
$ docker exec -it pg-lab psql -U dweb -d dweb_db

# 로컬 머신에서 실행 (psql 설치 필요)
$ psql -h localhost -p 5432 -U dweb -d dweb_db
```

psql 설치:
```terminal
# Ubuntu
$ sudo apt update && sudo apt install postgresql-client

# macOS
$ brew install postgresql
```

### psql 주요 명령어

| 명령어 | 설명 |
|--------|------|
| `\l` (소문자 L) | 모든 데이터베이스 목록 |
| `\dt` | 현재 DB의 테이블 목록 |
| `\dv` | 현재 DB의 뷰 목록 |
| `\d 테이블명` | 테이블 스키마(구조) 확인 |
| `\d+ 테이블명` | 테이블 상세 정보 확인 |
| `\c 데이터베이스명` | 다른 DB로 전환 |
| `\i 파일경로` | 파일에서 SQL 명령 실행 |
| `\q` | psql 종료 |
| `\! 명령어` | psql 안에서 쉘 명령 실행 (예: `\! clear`, `\! ls`) |
| `\du` | 사용자(Role) 목록 확인 |
| `\du+` | 사용자(Role) 상세 목록 확인 |
| `help` | 도움말 |

## 2. Database & Table 생성

### Database 생성 / 삭제

```sql
CREATE DATABASE [데이터베이스명];   -- 생성
DROP DATABASE [데이터베이스명];     -- 삭제
```

> 현재 접속 중인 DB는 삭제 불가. 다른 DB로 먼저 이동 후 삭제해야 함.
> ```sql
> \c postgres                    -- 기본 DB로 이동
> DROP DATABASE [데이터베이스명];
> ```

### Table 생성 / 삭제

```sql
-- 생성
CREATE TABLE [테이블명] (
    [컬럼명] [데이터타입],
    [컬럼명] [데이터타입],
    ...
);

-- 삭제 (테이블 + 데이터 모두 삭제)
DROP TABLE [테이블명];
```

> 테이블이 어떻게 생성되었는지 확인:
> ```sql
> \! pg_dump -s -t public.student_lists -d dweb_db -h localhost -p 5432 -U dweb
> ```

## 3. 데이터 삽입 — INSERT

```sql
-- 기본
INSERT INTO [테이블명] ([컬럼1], [컬럼2], [컬럼3])
VALUES ([값1], [값2], [값3]);

-- 컬럼 순서 변경 가능
INSERT INTO [테이블명] ([컬럼2], [컬럼1], [컬럼3])
VALUES ([값2], [값1], [값3]);

-- 컬럼 이름 생략 (테이블 컬럼 순서대로 입력)
INSERT INTO [테이블명]
VALUES ([값1], [값2], [값3]);

-- 일부 컬럼만 입력 (나머지는 NULL 또는 DEFAULT)
INSERT INTO [테이블명] ([컬럼1], [컬럼2])
VALUES ([값1], [값2]);
```

> 더미 데이터 생성 도구: [Mockaroo](https://www.mockaroo.com), [Generatedata](https://generatedata.com)

### INSERT ... SELECT

다른 테이블에서 조회한 결과를 그대로 INSERT:

```sql
INSERT INTO orders (email, product, qty)
SELECT email, 'Keyboard', 1
FROM source_users
WHERE email = 'alice@example.com';
```

## 4. 데이터 조회 — SELECT

### 기본 조회

```sql
SELECT * FROM [테이블명];                           -- 전체 컬럼
SELECT [컬럼1], [컬럼2] FROM [테이블명];            -- 특정 컬럼만
```

### AS — 컬럼 별칭

실제 테이블 구조는 바뀌지 않고 조회 결과에서만 이름이 바뀜.

```sql
SELECT [컬럼명] AS [별칭] FROM [테이블명];
SELECT 10*5 AS result;
```

- 별칭은 선택 사항
- 조회 결과에서만 표시되고 DB 구조에는 영향 없음 (일시적)
- 일반 컬럼은 AS 생략 시 원래 컬럼명 그대로 표시됨 (`SELECT name` → `name`)
- 계산식/함수는 AS 생략 시 의미없는 이름으로 표시됨
  - 계산식 : `SELECT 10*5` → `?column?`
  - 함수 : `SELECT AVG(price)` → `avg`
  - AS 사용 : `SELECT 10*5 AS result` → `result`

**쓰는 이유**
- 프론트엔드가 원하는 JSON 키 이름으로 맞춰줄 때 (`user_id` → `userId`)
- 계산식이나 함수 결과에 의미 있는 이름을 붙일 때
- JOIN 시 같은 이름의 컬럼이 충돌할 때 구분하기 위해

실무 활용:
```sql
-- 프론트엔드가 원하는 JSON 키 이름으로 맞춰줄 때
SELECT user_id AS "userId", created_at AS "createdAt" FROM users;

-- JOIN 시 컬럼 이름 충돌 방지 (뒤에서 등장)
```

### WHERE — 조건 필터링

조회할 데이터의 조건을 지정. 조건에 맞는 행만 반환.

1) 단순 조건
```sql
SELECT * FROM users WHERE status = 'active';

-- [결과]
-- ('alice', 'active'), ('bob', 'active')
```

2) OR — 조건 중 하나라도 맞으면
```sql
SELECT * FROM users WHERE status = 'active' OR status = 'pending';

-- [결과]
-- ('alice', 'active'), ('carol', 'pending')
```

3) AND — 조건 둘 다 맞아야
```sql
SELECT * FROM users WHERE status = 'active' AND age >= 20;

-- [결과]
-- ('alice', 'active', 25)
```

4) 괄호로 우선순위 묶기
```sql
SELECT * FROM users WHERE (status = 'active' OR status = 'pending') AND age >= 20;

-- [결과]
-- ('alice', 'active', 25), ('carol', 'pending', 22)
```

### LIKE — 패턴 검색

문자열에서 특정 패턴을 찾는 검색.

| 와일드카드 | 의미 | 상세 |
|-----------|------|------|
| `%` | 0개 이상의 임의 문자 | **위치**만 맞으면 됨. 개수 자유 |
| `_` | 정확히 1개의 임의 문자 | **위치 + 개수** 둘 다 정확히 맞아야 함 |

1) `%` — **위치**가 핵심 (개수 상관없음)
```sql
SELECT * FROM [테이블명] WHERE [컬럼명] LIKE 'a%';  -- a로 시작
SELECT * FROM [테이블명] WHERE [컬럼명] LIKE '%a';  -- a로 끝남
SELECT * FROM [테이블명] WHERE [컬럼명] LIKE '%a%'; -- a가 포함
SELECT * FROM [테이블명] WHERE [컬럼명] LIKE '%CA'; -- CA로 끝나는 값

-- [결과] LIKE 'a%'
-- ('alice'), ('a'), ('abcdefg')
-- ('balice') ❌ → % 앞 a가 없음

-- [결과] LIKE '%CA' → % 위치가 앞이니까 앞에 뭐가 몇 개 오든 상관없음
-- ('San Francisco, CA'), ('Los Angeles, CA'), ('San Diego, CA')
```

2) `_` — **위치 + 개수** 둘 다 정확히 맞아야 함
```sql
SELECT * FROM [테이블명] WHERE [컬럼명] LIKE 'a_';       -- a 뒤에 딱 1글자
SELECT * FROM [테이블명] WHERE [컬럼명] LIKE '_____@%';  -- @ 앞에 정확히 5글자

-- [결과] LIKE 'a_'
-- ('ab'), ('a1')
-- ('alice') ❌ → _ 1개인데 뒤에 4글자 / ('a') ❌ → _ 1개인데 뒤에 0글자

-- [결과] LIKE '_____@%' → _ 5개니까 @ 앞 글자 수가 정확히 5개여야 함, @ 뒤는 % 라서 상관없음
-- ('alice@gmail.com'), ('boost@naver.com'), ('hello@kakao.com')
-- ('al@gmail.com') ❌ → @ 앞이 2글자 / ('123456@gmail.com') ❌ → @ 앞이 6글자
```


### BETWEEN — 범위 검색

특정 범위 안에 있는 데이터를 찾는 검색.

| 특징 | 설명 |
|------|------|
| 끝값 | A, B 자기 자신 포함 (`>=`, `<=`) |
| 타입 | 숫자, 날짜, 문자열 |
| NULL | NULL 값은 범위에서 제외됨 |

1) 숫자 범위
```sql
SELECT * FROM [테이블명] WHERE age BETWEEN 20 AND 30;

-- [결과] 20 이상 30 이하 (양 끝값 포함)
-- ('alice', 25), ('bob', 20), ('carol', 30)
-- ('dave', 19) ❌ → 20 미만 / ('eve', 31) ❌ → 30 초과
```

2) 날짜 범위
```sql
SELECT * FROM [테이블명] WHERE birthday BETWEEN '1975-01-01' AND '1980-12-31';

-- [결과] 1975년 1월 1일 ~ 1980년 12월 31일 사이
-- ('1975-01-01'), ('1978-06-15'), ('1980-12-31')
-- ('1974-12-31') ❌ → 범위 이전 / ('1981-01-01') ❌ → 범위 이후
```

3) 문자열 범위
```sql
SELECT * FROM [테이블명] WHERE name BETWEEN 'A' AND 'M';

-- [결과] 알파벳 순서로 A ~ M 사이
-- ('Alice'), ('Bob'), ('Dave'), ('Mike')
-- ('Nathan') ❌ → N은 M 이후 / ('Zoe') ❌ → Z는 M 이후
```

> `>= A AND <= B` 랑 완전히 동일하지만 BETWEEN 이 더 짧고 읽기 편해서 씀.

### IN — 다중 값 검색

여러 개의 값 중 하나라도 일치하면 반환. OR를 여러 번 쓰는 것과 동일하지만 더 간결함.

| 특징 | 설명 |
|------|------|
| 동일 표현 | `= 값1 OR = 값2 OR = 값3` 과 완전히 동일 |
| 순서 | 상관없음 |
| NULL | NULL 값은 매칭 안 됨 |

1) 기본 사용
```sql
SELECT * FROM [테이블명] WHERE profession IN ('Student', 'Baker', 'Programmer');

-- [결과] 셋 중 하나라도 일치하면 반환
-- ('alice', 'Student'), ('bob', 'Baker'), ('carol', 'Programmer')
-- ('dave', 'Chef') ❌ → 목록에 없음
```

2) OR 대신 IN — 가독성 비교
```sql
-- OR 사용 (길고 반복적)
SELECT * FROM [테이블명]
WHERE profession = 'Student' OR profession = 'Baker' OR profession = 'Programmer';

-- IN 사용 (간결)
SELECT * FROM [테이블명]
WHERE profession IN ('Student', 'Baker', 'Programmer');
```

> 값이 많아질수록 OR보다 IN이 훨씬 읽기 편함.

### 비교 연산자

| 연산자 | 의미 |
|--------|------|
| `=` | 같다 |
| `<>` | 같지 않다 (표준) |
| `!=` | 같지 않다 (비표준, 대부분 지원) |
| `>` / `>=` | 크다 / 크거나 같다 |
| `<` / `<=` | 작다 / 작거나 같다 |

## 5. 고급 SELECT

### ORDER BY — 정렬

```sql
SELECT * FROM [테이블명] ORDER BY [컬럼명];              -- 오름차순 (기본)
SELECT * FROM [테이블명] ORDER BY [컬럼명] DESC;         -- 내림차순
SELECT * FROM [테이블명] ORDER BY [컬럼1], [컬럼2];      -- 여러 컬럼으로 정렬
```

### DISTINCT — 중복 제거

```sql
SELECT DISTINCT [컬럼명] FROM [테이블명];     -- 중복된 값은 한 번만 출력
```

### 집계 함수

```sql
SELECT MIN([컬럼명]) FROM [테이블명];
SELECT MAX([컬럼명]) FROM [테이블명];
SELECT AVG([컬럼명]) FROM [테이블명];
SELECT SUM([컬럼명]) FROM [테이블명];
SELECT SUM([컬럼1] * [컬럼2]) FROM [테이블명];  -- NULL이면 제외
SELECT COUNT([컬럼명]) FROM [테이블명];          -- NULL 제외하고 카운트
SELECT COUNT(*) FROM [테이블명];                 -- 전체 행 카운트
```

### COALESCE — NULL 대체

```sql
-- 값이 NULL이면 대체값으로 치환
SELECT [컬럼명], COALESCE([컬럼명], [대체값]) FROM [테이블명];

-- NULL을 0으로 치환해서 평균 계산
SELECT AVG(COALESCE([컬럼명], 0)) FROM [테이블명];
```

### NULLIF — 특정 값을 NULL로

```sql
SELECT NULLIF([값A], [값B]);   -- 같으면 NULL, 다르면 값A 반환

-- 0으로 나누기 방지
SELECT [컬럼1] / NULLIF([컬럼2], 0) FROM [테이블명];

-- COALESCE와 조합: 특정 값이면 대체값으로 처리
SELECT COALESCE(NULLIF([컬럼명], [특정값]), [대체값]) FROM [테이블명];
```

### GROUP BY — 그룹 집계

```sql
SELECT [컬럼명], COUNT(*) FROM [테이블명] GROUP BY [컬럼명];
SELECT [컬럼명], COUNT(*), SUM([컬럼명]) FROM [테이블명] GROUP BY [컬럼명] ORDER BY [컬럼명];
SELECT [컬럼명], AVG([컬럼명]) FROM [테이블명] GROUP BY [컬럼명];
```

### HAVING — 그룹 필터링

GROUP BY 이후 그룹 조건 필터링. WHERE는 행 단위, HAVING은 그룹 단위.

```sql
SELECT [컬럼명], SUM([컬럼명])
FROM [테이블명]
GROUP BY [컬럼명]
HAVING SUM([컬럼명]) >= [값];
-- 1. 컬럼 기준으로 그룹화
-- 2. 그룹별 합계 계산
-- 3. 조건을 만족하는 그룹만 반환
```


## 7. ON CONFLICT (충돌 처리)

### DO NOTHING — 중복 무시

```sql
INSERT INTO newsletter_users(email)
VALUES ('alice@example.com')
ON CONFLICT (email) DO NOTHING;
-- 이미 존재하면 에러 없이 무시
```

대량 삽입 시:
```sql
INSERT INTO newsletter_users(email)
SELECT email FROM staging_list
ON CONFLICT (email) DO NOTHING;
```

성공한 행만 반환:
```sql
INSERT INTO newsletter_users(email)
SELECT email FROM staging_list
ON CONFLICT (email) DO NOTHING
RETURNING email;
```

### DO UPDATE — UPSERT (없으면 INSERT, 있으면 UPDATE)

```sql
INSERT INTO products (product_id, product_name, stock)
VALUES (1, 'macbook', 5)
ON CONFLICT (product_id) DO UPDATE
SET stock = products.stock + EXCLUDED.stock;
-- product_id=1이 없으면 새로 INSERT
-- 있으면 기존 재고 + 새 입고량으로 UPDATE
-- EXCLUDED : INSERT하려고 했던 값
```

| 상황 | 문법 |
|------|------|
| 이미 있으면 무시 | `ON CONFLICT DO NOTHING` |
| 이미 있으면 갱신 | `ON CONFLICT DO UPDATE` |

## 8. 제약 조건 (Constraints)

### Primary Key (PK)

각 row를 유일하게 식별하는 컬럼. 중복·NULL 불가. 테이블당 1개.

```sql
CREATE TABLE orders (
    order_id BIGSERIAL PRIMARY KEY,  -- 자동 증가 정수 PK
    customer_id INT,
    order_date DATE,
    amount INT
);
```

> `BIGSERIAL` = 자동 증가하는 정수. `INT GENERATED ALWAYS AS IDENTITY`와 동일한 역할.

PK 나중에 추가:
```sql
ALTER TABLE orders ADD COLUMN order_id BIGSERIAL;
ALTER TABLE orders ADD PRIMARY KEY (order_id);
```

PK 제거:
```sql
ALTER TABLE orders DROP CONSTRAINT orders_pkey;
```

### UNIQUE

컬럼 값 중복 불가 제약. NULL은 허용. 테이블당 여러 개 가능.

```sql
ALTER TABLE students ADD CONSTRAINT unique_student_email UNIQUE (email);
ALTER TABLE students ADD UNIQUE (email);           -- 이름 생략 가능 (자동 생성)
ALTER TABLE students DROP CONSTRAINT unique_student_email;  -- 제거
```

| | PK | UNIQUE |
|--|:--:|:--:|
| 중복 금지 | ✅ | ✅ |
| NULL 허용 | ❌ | ✅ |
| 테이블당 개수 | 1개 | 여러 개 |
| 자동 인덱스 | ✅ | ✅ |
| 역할 | 행의 정체성 | 업무 규칙 |

### CHECK

비즈니스 로직을 DB 레벨에서 강제.

```sql
-- 나이 제한
CREATE TABLE users (
    user_id INT,
    age INT,
    CONSTRAINT check_age_range CHECK (age BETWEEN 0 AND 120)
);

-- 성별 값 제한
CONSTRAINT check_gender CHECK (gender IN ('M', 'F'))

-- 금액 음수 불가
CONSTRAINT check_amount_positive CHECK (amount >= 0)
```

나중에 추가:
```sql
ALTER TABLE exam_scores
ADD CONSTRAINT check_score_range CHECK (score BETWEEN 0 AND 100);
```

> 앱에서만 검사하면 버그나 우회로 뚫릴 수 있음. DB에서 검사하면 어떤 경로로 들어와도 규칙이 강제됨.

## 9. 다중 테이블 설계

### 단일 테이블의 문제점

모든 정보를 하나의 테이블에 넣으면 3가지 이상 현상 발생:

| 이상 현상 | 설명 |
|----------|------|
| **갱신 이상** (Update Anomaly) | 중복 저장된 데이터 일부만 수정하면 불일치 발생 |
| **삭제 이상** (Delete Anomaly) | 데이터 삭제 시 의도치 않게 다른 정보도 함께 삭제 |
| **삽입 이상** (Insert Anomaly) | 특정 정보만 독립적으로 삽입 불가 |

예: 사용자 정보 + 주문 정보가 한 테이블에 있으면, Alice의 마지막 주문을 삭제하면 Alice 고객 정보 자체가 사라짐.

### 정규화 (Normalization)

중복을 제거하고 데이터를 여러 테이블로 분리하는 과정.

### Foreign Key (FK)

한 테이블의 컬럼이 다른 테이블의 PK를 참조하는 제약 조건. 두 테이블 간의 관계를 설정하고 **참조 무결성**을 보장.

```sql
CREATE TABLE orders (
    order_id   SERIAL PRIMARY KEY,
    user_email TEXT NOT NULL REFERENCES users(user_email),  -- FK
    product    TEXT NOT NULL,
    qty        INT NOT NULL
);
```

- **참조 무결성** : FK가 가리키는 값이 반드시 부모 테이블에 존재해야 함
- FK가 없으면 → 주문은 있는데 고객이 없는 **고아 데이터(orphan row)** 발생
- 일대다 관계에서 FK는 항상 **N(다수) 쪽**에 위치

### 다중 테이블 INSERT

```sql
-- 고객 먼저 등록 (테이블 분리로 주문 없이 저장 가능)
INSERT INTO users(user_email, user_name, user_phone)
VALUES ('alice@inu.ac.kr', 'Alice', '010-1111-2222');

-- 주문 등록 (user_email을 조회해서 FK로 연결)
INSERT INTO orders(user_email, product, qty)
SELECT user_email, 'Keyboard', 1
FROM users
WHERE user_email = 'alice@inu.ac.kr';
```

### ON DELETE 옵션

부모 데이터 삭제 시 자식 데이터 처리 방법:

| 옵션 | 동작 |
|------|------|
| `RESTRICT` (기본값) | 자식이 참조 중이면 부모 삭제 막음 |
| `CASCADE` | 부모 삭제 시 자식도 함께 삭제 |

```sql
-- CASCADE로 변경
ALTER TABLE orders DROP CONSTRAINT orders_user_email_fkey;
ALTER TABLE orders ADD CONSTRAINT orders_user_email_fkey
    FOREIGN KEY (user_email) REFERENCES users(user_email)
    ON DELETE CASCADE;
```

## 10. JOIN

정규화로 분리된 테이블을 다시 연결해서 조회.

### INNER JOIN

양쪽 테이블에 모두 대응되는 행이 있을 때만 결과에 포함.

```sql
SELECT u.user_email, u.user_name, o.order_id, o.product, o.qty
FROM users u
INNER JOIN orders o ON u.user_email = o.user_email
ORDER BY o.order_id;
-- 주문이 없는 사용자는 결과에 포함되지 않음
```

### LEFT JOIN

왼쪽 테이블의 모든 행을 출력. 오른쪽에 없는 값은 NULL로 반환.

```sql
SELECT u.user_email, u.user_name, o.order_id, o.product
FROM users u
LEFT JOIN orders o ON u.user_email = o.user_email
ORDER BY o.order_id;
-- 주문이 없는 사용자도 포함 (주문 컬럼은 NULL)
```

### CROSS JOIN (주의)

JOIN 조건 없이 조회하면 두 테이블의 모든 행을 서로 다 조합함.

```sql
-- 아래 두 쿼리는 동일 (의도하지 않은 결과 주의)
SELECT * FROM users u, orders o;
SELECT * FROM users u CROSS JOIN orders o;
```

## 11. View (가상 테이블)

실제 데이터를 저장하지 않는 가상 테이블. 조회 시점에 원본 테이블을 다시 읽어서 결과를 생성함 (Query Rewrite).

```sql
-- 뷰 생성
CREATE VIEW [뷰이름] AS
SELECT [컬럼들]
FROM [테이블]
WHERE [조건];

-- 뷰 조회 (일반 테이블처럼 사용)
SELECT * FROM [뷰이름];

-- 뷰 재정의 (있으면 교체, 없으면 생성)
CREATE OR REPLACE VIEW [뷰이름] AS
SELECT [컬럼들] FROM [테이블];
```

**쓰는 이유 3가지**

1. **복잡한 쿼리 재사용** — 복잡한 JOIN 쿼리를 단순한 명령 하나로 대체
```sql
CREATE VIEW order_summary AS
SELECT u.user_email, u.user_name, o.order_id, o.product, o.qty, o.ordered_at
FROM users u
LEFT JOIN orders o ON u.user_email = o.user_email
ORDER BY o.order_id;

SELECT * FROM order_summary;  -- JOIN 없이 조회 가능
```

2. **DB 구조 변경 시 앱 영향 최소화** — 내부 쿼리가 바뀌어도 앱은 뷰 이름만 바라보면 됨
```sql
-- JOIN 조건 변경 시 뷰만 수정하면 됨
CREATE OR REPLACE VIEW order_summary AS
SELECT u.user_email, u.user_name, o.order_id, o.product, o.qty, o.ordered_at
FROM users u
INNER JOIN orders o ON u.user_email = o.user_email
ORDER BY o.order_id;
```

3. **민감한 컬럼 숨기기** — 특정 컬럼(phone 등)을 제외한 뷰를 만들어 접근 제한
```sql
CREATE VIEW customer_view AS
SELECT user_id, email, name, created_at  -- phone 컬럼 제외
FROM customers;
```

> View는 INSERT/UPDATE/DELETE 용도로 쓰지 않음. 조회 전용.

## 12. Access Control (권한 관리)

PostgreSQL은 Role 기반으로 사용자와 권한을 관리함.

### 사용자(Role) 생성

```sql
CREATE ROLE [사용자명] LOGIN PASSWORD '[비밀번호]';
-- CREATE USER [사용자명] PASSWORD '[비밀번호]'; 와 동일
```

### GRANT — 권한 부여

Role을 만들고 → Role에 권한을 부여하고 → 사용자에게 Role을 부여하는 3단계 구조.

```sql
-- 1. Role 생성
CREATE ROLE readonly_role;
CREATE ROLE write_role;

-- 2. Role에 권한 부여
-- readonly_role: SELECT만 허용
GRANT USAGE ON SCHEMA public TO readonly_role;
GRANT SELECT ON [테이블명] TO readonly_role;

-- write_role: SELECT + INSERT + UPDATE + DELETE 허용
GRANT USAGE ON SCHEMA public TO write_role;
GRANT USAGE, SELECT ON ALL SEQUENCES IN SCHEMA public TO write_role;
GRANT SELECT, INSERT, UPDATE, DELETE ON ALL TABLES IN SCHEMA public TO write_role;

-- 3. 사용자에게 Role 부여
GRANT readonly_role TO [사용자명];
GRANT write_role TO [사용자명];
```

### REVOKE — 권한 취소

```sql
-- GRANT 반대. TO 대신 FROM 사용
REVOKE readonly_role FROM [사용자명];
REVOKE write_role FROM [사용자명];
```

| 구분 | 명령 |
|------|------|
| 권한 부여 | `GRANT [role] TO [user]` |
| 권한 취소 | `REVOKE [role] FROM [user]` |

> View와 조합하면 강력함 — 원본 테이블 권한은 막고, 민감 컬럼이 제거된 View에만 접근 권한을 부여할 수 있음.

## 핵심 요약

```
DDL  : CREATE, DROP (구조)
DML  : INSERT, SELECT, UPDATE, DELETE (데이터)

SELECT 절 순서
  SELECT → FROM → WHERE → GROUP BY → HAVING → ORDER BY

제약 조건
  PK      : 행 식별, 중복·NULL 불가, 1개
  UNIQUE  : 중복 불가, NULL 허용, 여러 개
  CHECK   : 비즈니스 로직 강제
  FK      : 테이블 간 관계, 참조 무결성 보장

단일 테이블 문제
  갱신 이상 / 삭제 이상 / 삽입 이상 → 정규화로 해결

JOIN
  INNER JOIN : 양쪽 모두 있는 행만
  LEFT JOIN  : 왼쪽 테이블 전체 + 오른쪽 없으면 NULL
  CROSS JOIN : 모든 조합 (주의)

ON CONFLICT
  DO NOTHING : 중복 무시
  DO UPDATE  : UPSERT (없으면 INSERT, 있으면 UPDATE)

View
  가상 테이블. 실제 데이터 저장 안 함
  복잡한 쿼리 재사용 / 구조 변경 대응 / 컬럼 숨기기

Access Control
  GRANT  : 권한 부여 (Role → User)
  REVOKE : 권한 취소
```
