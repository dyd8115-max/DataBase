# Database Internals — Storage & Indexing

## 1. 세상에서 가장 단순한 DB

```bash
db_set () {
    echo "$1,$2" >> database
}
db_get () {
    grep "^$1," database | sed -e "s/^$1,//" | tail -n 1
}
```

- `db_set` : key-value 쌍을 파일 끝에 **append**
- `db_get` : 파일 전체를 grep → **가장 마지막** 값 반환 (tail -n 1)

| 연산 | 시간복잡도 | 이유 |
|------|-----------|------|
| `db_set` | O(1) | 그냥 파일 끝에 붙임 |
| `db_get` | O(n) | 파일 전체 scan |

> 같은 key로 `db_set`을 여러 번 해도 overwrite가 아니라 append.  
> 조회 시 `tail -n 1`으로 최신 값만 가져옴.

## 2. 인덱스가 왜 필요한가

### Full Table Scan의 문제

```sql
SELECT * FROM student WHERE student_id = 2021984;
```

인덱스 없으면 → 테이블 전체를 처음부터 끝까지 읽음 = **O(n)**

### 인덱스의 트레이드오프

```
읽기 성능  ↑  (원하는 row 빠르게 찾음)
쓰기 성능  ↓  (INSERT / UPDATE / DELETE 시 인덱스도 같이 갱신)
```

> 인덱스는 쿼리 성능에만 영향을 줌. 데이터 자체는 바뀌지 않음.

---

## 3. 인덱스 종류

### 3-1. Secondary Index (보조 인덱스)

PK, UNIQUE 외의 컬럼에 별도로 생성하는 인덱스.

```sql
CREATE INDEX idx_users_email ON users(email);
```

내부 구조:

```
key (email)          | 물리 주소
---------------------|----------
alice@example.com    | 0xAB12
bob@example.com      | 0xFF09
carol@example.com    | 0x9C34
```

- key : 인덱스 컬럼 값
- value : 해당 row의 물리 주소(포인터)

> Secondary Index가 많을수록 INSERT / UPDATE / DELETE 비용 증가.

---

### 3-2. Composite Index (복합 인덱스)

여러 컬럼을 묶어서 하나의 인덱스로 만듦.

```sql
CREATE INDEX idx_orders_lookup ON orders(user_id, status, created_at);
```

인덱스 내부는 **(user_id, status, created_at)** 순서로 정렬됨:

```
(1, 'PAID',    2025-02-01) → 주소
(1, 'PAID',    2025-02-02) → 주소
(1, 'PENDING', 2025-01-30) → 주소
(2, 'PAID',    2025-02-03) → 주소
(2, 'CANCEL',  2025-01-29) → 주소
```

#### 인덱스 활용 가능 여부 (Leftmost Prefix Rule)

| WHERE 조건 | 인덱스 사용 |
|-----------|-----------|
| `user_id = 10` | ✅ |
| `user_id = 10 AND status = 'PAID'` | ✅ |
| `user_id = 10 AND status = 'PAID' AND created_at >= '2025-02-01'` | ✅ |
| `status = 'PAID'` | ❌ (첫 번째 컬럼 빠짐) |
| `created_at >= '2025-02-01'` | ❌ (첫 번째 컬럼 빠짐) |
| `status = 'PAID' AND created_at > '2025-02-01'` | ❌ (첫 번째 컬럼 빠짐) |

> 복합 인덱스는 **왼쪽 컬럼부터 순서대로** 써야 인덱스를 탐. 중간 컬럼이 빠지면 그 이후 컬럼도 인덱스 못 씀.

## 4. PostgreSQL 인덱스 타입

```sql
CREATE INDEX idx_users_email ON users USING btree (email);
```

| 타입 | 특징 | 용도 |
|------|------|------|
| **B-Tree** | 기본값, 정렬된 트리 구조 | 범위 조회, 정렬, 일반 등호 |
| **Hash** | 해시 테이블 | `=` 동등 비교만 |
| **GIN** | Inverted Index | 배열, JSONB, Full-text search |
| **GiST** | Generalized Tree | 공간 데이터, 기하 연산 |
| **SP-GiST** | Space-partitioned Tree | 비균형 구조 데이터 |
| **BRIN** | Block Range Index | 순서대로 쌓이는 대용량 테이블 (append-only) |

> 대부분의 경우 B-Tree로 충분. 특수한 쿼리 패턴이 있을 때만 다른 타입 고려.

## 핵심 요약

```
인덱스 = 읽기 빠르게, 쓰기 느리게
복합 인덱스 = 왼쪽 컬럼부터 순서대로 써야 효과 있음
인덱스 없으면 = Full Table Scan (O(n))
인덱스 있으면 = 포인터로 바로 접근
```
