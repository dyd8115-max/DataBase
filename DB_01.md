# Database Internals — Storage & Indexing

---


### 쉘 (Shell)

- 사용자가 운영체제에 명령을 내리는 인터페이스 (터미널 실행 환경)
- **쉘 파일** : 쉘 명령들을 모아놓은 파일. 반복 작업이나 함수를 정의해두고 재사용 가능 (확장자 `.sh`)

### 쉘 파일 실행 방법

```terminal
$ chmod +x simpleDB.sh    # 실행 권한 부여 (최초 1회)
$ source simpleDB.sh      # 쉘 파일을 현재 터미널에 로드 → 함수 사용 가능
```

> `source` 없이 `./simpleDB.sh`로만 실행하면 함수가 현재 터미널 세션에 남지 않아 `db_set`, `db_get`을 쓸 수 없음.

---

## 1. 세상에서 가장 단순한 DB

`simpleDB.sh` 쉘 파일 내부:

```sh
#!/bin/bash
db_set () {
    echo "$1,$2" >> database  # $1(key),$2(value) 형태로 현재 디렉토리의 database 파일 끝에 append
}
db_get () {
    grep "^$1," database |    # database 파일에서 key,로 시작하는 줄만 필터링
    sed -e "s/^$1,//" |       # 필터링된 줄에서 key, 부분 제거 → value만 남김
    tail -n 1                 # 여러 줄 중 마지막 줄(최신 값)만 반환
}
```

### db_set — 데이터 저장

> overwrite 없이 **무조건 append** — 같은 key로 여러 번 저장해도 기존 데이터는 유지되고 파일에 계속 쌓임

---

### db_get — 데이터 조회

파이프(`|`)로 3단계를 순서대로 처리함. `$ db_get 1` 실행 시 내부 흐름:

```
[database 파일 전체]
1,{"major":"CS"}    ← grep: key=1로 시작하는 줄 필터링
2,{"major":"EE"}
1,{"major":"AI"}    ← grep: 이 줄도 필터링
1,{"major":"Data"}  ← grep: 이 줄도 필터링

[sed 후 → key, 부분 제거]
{"major":"CS"}
{"major":"AI"}
{"major":"Data"}

[tail -n 1 → 마지막 값만 반환]
{"major":"Data"}
```

### 시간복잡도

| 연산 | 시간복잡도 | 이유 |
|------|-----------|------|
| `db_set` | O(1) | 파일 끝에 append만 함 |
| `db_get` | O(n) | 파일 전체 scan |

> `db_get`이 O(n)이라는 건 데이터가 100만 건이면 최대 100만 번 읽는다는 뜻.  
> 특정 key의 value를 효율적으로 찾으려면 다른 자료구조가 필요 → **Index**

## 2. Index

**인덱싱(Indexing)** : 특정 컬럼에 인덱스를 생성하고 관리하는 행위. 인덱싱을 하면 해당 컬럼으로 조회할 때 Full Scan 없이 빠르게 찾을 수 있음.

### 왜 필요한가 — Full Table Scan의 문제

```sql
SELECT * FROM student WHERE student_id = 2021984;
```

인덱스가 없으면 테이블 전체를 처음부터 끝까지 읽음 = **O(n)**

```
student_id  name  major
2021231     Kim   CS
2020103     Lee   AI
2019203     Park  Data
2020123     Han   EE
2023034     Choi  Bio
2018395     Kim   Physics
2022021     Song  Mechanical
2021984     Go    Energy      ← 여기까지 전부 읽어야 찾음
2024484     Jih   Industrial
2022347     jin   Safety
2025932     Son   Materials
```

### 인덱스 적용 시

student_id를 기준으로 **정렬된 별도 구조**를 만들고, 각 값에 해당하는 row의 **물리 주소(포인터)**를 저장.

```
student_id (정렬) | Pointer
------------------|--------
2018395           | →
2019203           | →
2020103           | →
2020123           | →
2021231           | →
2021984           | → (바로 접근)
...
```

인덱스에서 `2021984`를 찾으면 포인터로 **바로 해당 row에 접근** → Full Scan 불필요.

### 트레이드오프

```
읽기 성능  ↑  (인덱스 → 포인터 → 바로 접근)
쓰기 성능  ↓  (INSERT / UPDATE / DELETE 시 인덱스도 같이 갱신해야 함)
```

- 인덱스는 데이터 자체를 바꾸지 않음. **쿼리 성능에만** 영향을 줌
- 인덱스가 많을수록 읽기는 빠르지만 쓰기는 점점 느려짐

> **쿼리(Query)** : DB에 보내는 질문 또는 명령. `SELECT`, `INSERT`, `UPDATE`, `DELETE` 등

### 인덱스의 한계

인덱스가 걸린 컬럼으로만 빠른 조회가 가능함.

```sql
-- student_id에 인덱스가 있어도 name으로 조회하면 Full Scan
SELECT * FROM student WHERE name = 'Han';
```

name 컬럼에 별도 인덱스를 만들지 않으면 여전히 Full Table Scan.

- **저장 공간 추가 사용** : 인덱스는 원본 데이터와 별도로 저장되는 구조라 그만큼 디스크를 더 차지함. 인덱스가 많을수록 저장 공간 부담도 커짐

## 3. 인덱스 종류

### 3-1. Secondary Index (보조 인덱스)

PK, UNIQUE가 아닌 컬럼에 **별도로 생성**하는 인덱스.  
PK와 UNIQUE 컬럼에는 인덱스가 **자동 생성**됨.

> **PK (Primary Key, 기본키)** : 각 row를 유일하게 식별하는 컬럼. 중복·NULL 불가. 테이블당 1개  
> **UNIQUE** : 해당 컬럼의 값이 중복되면 안 된다는 제약조건. 테이블당 여러 개 가능  
> 둘 다 조회에 자주 쓰이기 때문에 인덱스가 자동 생성됨

```sql
CREATE TABLE users (
    id    INT,
    email TEXT,
    name  TEXT
);

-- email 컬럼에 보조 인덱스 생성
CREATE INDEX idx_users_email ON users(email);
```

내부 구조:

```
key (email)          | 물리 주소
---------------------|----------
alice@example.com    | 0xAB12
bob@example.com      | 0xFF09
carol@example.com    | 0x9C34
```

- **key** : 인덱스 컬럼 값 (email)
- **value** : 해당 row의 물리 주소 (포인터)
- email로 검색하면 인덱스에서 주소를 찾아 바로 접근

> Secondary Index가 늘어날수록 INSERT / UPDATE / DELETE 때마다 갱신해야 할 인덱스가 많아짐 → 쓰기 비용 증가.

---

### 3-2. Composite Index (복합 인덱스)

여러 컬럼을 묶어서 **하나의 인덱스**로 만듦.  
컬럼을 각각 따로 인덱스 만드는 것과 다르게, 여러 조건을 **동시에** 빠르게 처리할 수 있음.

```sql
CREATE INDEX idx_orders_lookup ON orders(user_id, status, created_at);
```

인덱스 내부는 **(user_id → status → created_at)** 순서로 **정렬**됨:

```
(1, 'PAID',    2025-02-01) → 주소
(1, 'PAID',    2025-02-02) → 주소
(1, 'PENDING', 2025-01-30) → 주소
(2, 'PAID',    2025-02-03) → 주소
(2, 'CANCEL',  2025-01-29) → 주소
```

#### Leftmost Prefix Rule

복합 인덱스는 **왼쪽 컬럼부터 순서대로** 매칭되는 구조.  
중간 컬럼이 WHERE에 빠지면 그 이후 컬럼은 인덱스를 탈 수 없음.

| WHERE 조건 | 인덱스 사용 | 이유 |
|-----------|:----------:|------|
| `user_id = 10` | ✅ | 첫 번째 컬럼 매칭 |
| `user_id = 10 AND status = 'PAID'` | ✅ | 첫 번째 + 두 번째 매칭 |
| `user_id = 10 AND status = 'PAID' AND created_at >= '2025-02-01'` | ✅ | 전체 매칭 |
| `status = 'PAID'` | ❌ | 첫 번째 컬럼(user_id) 빠짐 |
| `created_at >= '2025-02-01'` | ❌ | 첫 번째 컬럼(user_id) 빠짐 |
| `status = 'PAID' AND created_at > '2025-02-01'` | ❌ | 첫 번째 컬럼(user_id) 빠짐 |

활용 예시:

```sql
-- 이 쿼리는 인덱스 (user_id, status, created_at) 전체를 활용
SELECT *
FROM orders
WHERE user_id = 10
  AND status = 'PAID'
  AND created_at >= '2025-02-01';
```

## 4. PostgreSQL 인덱스 타입

PostgreSQL은 `USING` 키워드로 인덱스 타입을 지정함. 생략하면 기본값 B-Tree.

```sql
CREATE INDEX idx_users_email ON users USING btree (email);
```

| 타입 | 내부 구조 | 주요 용도 |
|------|----------|----------|
| **B-Tree** | 정렬된 트리 | 범위 조회(`<`, `>`, `BETWEEN`), 정렬, 등호(`=`) — **기본값** |
| **Hash** | 해시 테이블 | `=` 동등 비교만. 범위 조회 불가 |
| **GIN** | Inverted Index | 배열, JSONB, Full-text search |
| **GiST** | Generalized Tree | 공간 데이터, 기하 연산 |
| **SP-GiST** | Space-partitioned Tree | 비균형 구조 데이터 (전화번호, IP 등) |
| **BRIN** | Block Range Index | 순서대로 append되는 대용량 테이블. 용량 매우 작음 |

> 대부분의 경우 B-Tree로 충분. 특수한 쿼리 패턴(배열, 공간, 대용량 로그)이 있을 때만 다른 타입 고려.

---

## 5. 인덱스 내부 자료구조

인덱스가 빠른 이유는 내부적으로 효율적인 자료구조를 사용하기 때문.

| 자료구조 | 특징 | 사용처 |
|----------|------|--------|
| **B-Tree** | 정렬된 트리. 범위 조회·정렬·등호 전부 가능 | DB 인덱스 표준 (MySQL, PostgreSQL 기본값) |
| **Hash Table** | key를 해시함수로 변환 → 바로 주소 접근. `=` 비교만 빠름, 범위 조회 불가 | PostgreSQL Hash 인덱스 |
| **LSM Tree** | 쓰기에 특화된 트리 구조. 읽기보다 쓰기가 압도적으로 많은 환경에 유리 | Cassandra, RocksDB |

> 일반적인 DB 인덱스는 대부분 **B-Tree** 기반. 나머지는 특수한 상황에서만 사용.

## 핵심 요약

```
단순 DB (append-only)
  db_set → O(1), db_get → O(n)
  데이터 많아질수록 조회 느려짐 → Index 필요

Index
  읽기 빠르게 (포인터로 바로 접근)
  쓰기 느리게 (데이터 변경 시 인덱스도 같이 갱신)
  인덱스 걸린 컬럼으로만 효과 있음

Secondary Index
  PK/UNIQUE 외 컬럼에 별도 생성
  많을수록 쓰기 비용 증가

Composite Index (복합 인덱스)
  Leftmost Prefix Rule — 왼쪽 컬럼부터 순서대로 써야 효과 있음
  중간 컬럼 빠지면 이후 컬럼도 인덱스 못 씀

PostgreSQL 인덱스 타입
  기본값     → B-Tree (범위, 정렬, 등호)
  등호만     → Hash
  배열/JSONB → GIN
  공간 데이터 → GiST
  대용량 로그 → BRIN
```
