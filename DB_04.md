# SQL with PostgreSQL — 설계 / 관리

## 1. 제약 조건 (Constraints)

### Primary Key (PK)

각 row를 유일하게 식별하는 컬럼. 중복·NULL 불가. 테이블당 1개.

```sql
CREATE TABLE orders (
    order_id BIGSERIAL PRIMARY KEY,  -- 자동 증가 정수 PK
    customer_id INT,
    order_date DATE,
    amount INT
);
```

> `BIGSERIAL` = 자동 증가하는 정수. `INT GENERATED ALWAYS AS IDENTITY`와 동일한 역할.

**자동 증가 방식 3가지 — 아래 세 줄은 같은 의미**

```sql
order_id SERIAL PRIMARY KEY
order_id INT NOT NULL DEFAULT nextval('orders_order_id_seq')
order_id INT GENERATED ALWAYS AS IDENTITY PRIMARY KEY
```

| 방식 | 내부 sequence | 수동 값 입력 |
|------|:---:|:---:|
| `SERIAL` | ✅ | ✅ |
| `DEFAULT nextval` | ✅ | ✅ |
| `IDENTITY ALWAYS` | ✅ | ❌ |

PK 나중에 추가 / 제거:
```sql
ALTER TABLE orders ADD COLUMN order_id BIGSERIAL;
ALTER TABLE orders ADD PRIMARY KEY (order_id);

ALTER TABLE orders DROP CONSTRAINT orders_pkey;  -- 제거
```

---

### UNIQUE

컬럼 값 중복 불가 제약. NULL은 허용. 테이블당 여러 개 가능.

```sql
ALTER TABLE students ADD CONSTRAINT unique_student_email UNIQUE (email);
ALTER TABLE students ADD UNIQUE (email);                        -- 이름 생략 가능 (자동 생성)
ALTER TABLE students DROP CONSTRAINT unique_student_email;      -- 제거
```

**PK vs UNIQUE**

| | PK | UNIQUE |
|--|:--:|:--:|
| 중복 금지 | ✅ | ✅ |
| NULL 허용 | ❌ | ✅ |
| 테이블당 개수 | 1개 | 여러 개 |
| 자동 인덱스 | ✅ | ✅ |
| 역할 | 행의 정체성 | 업무 규칙 |

---

### CHECK

비즈니스 로직을 DB 레벨에서 강제.

```sql
-- 나이 제한
CREATE TABLE users (
    user_id INT,
    age INT,
    CONSTRAINT check_age_range CHECK (age BETWEEN 0 AND 120)
);

-- 성별 값 제한
CONSTRAINT check_gender CHECK (gender IN ('M', 'F'))

-- 금액 음수 불가
CONSTRAINT check_amount_positive CHECK (amount >= 0)
```

나중에 추가:
```sql
ALTER TABLE exam_scores
ADD CONSTRAINT check_score_range CHECK (score BETWEEN 0 AND 100);
```

> 앱에서만 검사하면 버그나 우회로 뚫릴 수 있음. DB에서 검사하면 어떤 경로로 들어와도 규칙이 강제됨.

## 2. 다중 테이블 설계

### 단일 테이블의 문제점

모든 정보를 하나의 테이블에 넣으면 3가지 이상 현상 발생:

| 이상 현상 | 설명 |
|----------|------|
| **갱신 이상** (Update Anomaly) | 중복 저장된 데이터 일부만 수정하면 불일치 발생 |
| **삭제 이상** (Delete Anomaly) | 데이터 삭제 시 의도치 않게 다른 정보도 함께 삭제 |
| **삽입 이상** (Insert Anomaly) | 특정 정보만 독립적으로 삽입 불가 |

```sql
-- [원본 데이터] orders_one_table (사용자 정보 + 주문 정보가 한 테이블에 존재)
-- order_id | user_email       | user_name | user_phone    | product  | qty
--        1 | alice@inu.ac.kr  | Alice     | 010-1111-2222 | Keyboard | 1
--        2 | alice@inu.ac.kr  | Alice     | 010-1111-2222 | Mouse    | 2
--        3 | bob@inu.ac.kr    | Bob       | 010-3333-4444 | Monitor  | 1

-- 갱신 이상: Keyboard 주문만 전화번호 변경 → Alice 행마다 다른 번호 저장됨
UPDATE orders_one_table SET user_phone = '010-9999-4444' WHERE product = 'Keyboard';
-- order_id 1: 010-9999-4444 / order_id 2: 010-1111-2222 → 불일치 발생

-- 삭제 이상: Alice 주문 전부 삭제 → Alice 고객 정보 자체가 사라짐
DELETE FROM orders_one_table WHERE user_email = 'alice@inu.ac.kr';

-- 삽입 이상: 주문 없이 고객만 등록 불가
INSERT INTO orders_one_table (user_email, user_name, user_phone)
VALUES ('carol@inu.ac.kr', 'Carol', '010-0000-0101');  -- ❌ 에러
-- 억지로 넣으면 가짜 데이터가 섞여 통계 오류 발생
INSERT INTO orders_one_table (user_email, user_name, user_phone, product, qty)
VALUES ('carol@inu.ac.kr', 'Carol', '010-0000-0101', '주문없음', 0);
```

> **정규화 (Normalization)** : 중복을 제거하고 데이터를 여러 테이블로 분리해 이상 현상을 해결하는 과정.

### 정규형 (Normal Form)

릴레이션이 정규화된 정도. 차수가 높아질수록 제약조건이 엄격해짐.

**함수 종속성 (Functional Dependency)**

컬럼 A의 값이 결정되면 컬럼 B의 값도 결정되는 관계. `A → B` 로 표기.

```
student_id → student_name        (학번이 정해지면 이름도 결정)
subject_id → subject_name        (과목 코드가 정해지면 과목명도 결정)
(student_id, subject_id) → score (학번 + 과목 코드가 정해지면 점수가 결정)
```

**1NF — 원자값 (Atomic Value)**

모든 컬럼의 값이 더 이상 쪼갤 수 없는 단일 값이어야 함.

```
❌ 1NF 불만족
student_id | name  | subjects
1001       | Alice | Database, OS, Network   ← 한 셀에 여러 값

✅ 1NF 만족
student_id | name  | subject
1001       | Alice | Database
1001       | Alice | OS
1001       | Alice | Network
```

**2NF — 부분 함수 종속 제거**

1NF를 만족하면서, 기본키가 복합키일 때 일부 컬럼만으로 결정되는 컬럼이 없어야 함.

```
❌ 2NF 불만족 (복합 PK: student_id + subject_id)
student_id | subject_id | score | student_name | subject_name
1001       | DB01       | 85    | Alice        | Database
1001       | OS01       | 90    | Alice        | OS

종속 관계:
  student_id ──────────────────→ student_name  (부분 종속 ❌)
  subject_id ──────────────────→ subject_name  (부분 종속 ❌)
  (student_id, subject_id) ────→ score         (완전 종속 ✅)

✅ 2NF 만족 — 부분 종속 컬럼을 별도 테이블로 분리

  [students]            [subjects]            [scores]
  student_id (PK)       subject_id (PK)       student_id (FK) ──→ students
  student_name          subject_name          subject_id (FK) ──→ subjects
                                              score
```

### Foreign Key (FK)

한 테이블의 컬럼이 다른 테이블의 PK를 참조하는 제약 조건. 두 테이블 간의 관계를 설정하고 **참조 무결성**을 보장.

```sql
CREATE TABLE orders (
    order_id   SERIAL PRIMARY KEY,
    user_email TEXT NOT NULL REFERENCES users(user_email),  -- FK
    product    TEXT NOT NULL,
    qty        INT NOT NULL
);
```

- **참조 무결성** : FK가 가리키는 값이 반드시 부모 테이블에 존재해야 함
- FK가 없으면 → 주문은 있는데 고객이 없는 **고아 데이터(orphan row)** 발생
- 일대다 관계에서 FK는 항상 **N(다수) 쪽**에 위치

### 다중 테이블 INSERT

```sql
-- [원본 데이터] users 테이블 (비어있음)
-- user_email | user_name | user_phone

-- 고객 먼저 등록 (테이블 분리로 주문 없이 저장 가능)
INSERT INTO users(user_email, user_name, user_phone)
VALUES ('alice@inu.ac.kr', 'Alice', '010-1111-2222');
-- user_email       | user_name | user_phone
-- alice@inu.ac.kr  | Alice     | 010-1111-2222

-- 주문 등록 (user_email을 조회해서 FK로 연결)
INSERT INTO orders(user_email, product, qty)
SELECT user_email, 'Keyboard', 1
FROM users
WHERE user_email = 'alice@inu.ac.kr';
-- user_email       | product  | qty
-- alice@inu.ac.kr  | Keyboard | 1
```

### ON DELETE 옵션

부모 데이터 삭제 시 자식 데이터 처리 방법:

| 옵션 | 동작 |
|------|------|
| `RESTRICT` (기본값) | 자식이 참조 중이면 부모 삭제 막음 |
| `CASCADE` | 부모 삭제 시 자식도 함께 삭제 |

```sql
-- RESTRICT(기본값): alice 삭제 시도 → 에러
DELETE FROM users WHERE user_email = 'alice@inu.ac.kr';  -- ❌ 자식 행이 참조 중이라 막힘

-- CASCADE로 변경
ALTER TABLE orders DROP CONSTRAINT orders_user_email_fkey;
ALTER TABLE orders ADD CONSTRAINT orders_user_email_fkey
    FOREIGN KEY (user_email) REFERENCES users(user_email)
    ON DELETE CASCADE;

-- CASCADE 확인: alice 삭제 시 orders의 alice 주문도 함께 삭제됨
DELETE FROM users WHERE user_email = 'alice@inu.ac.kr';  -- ✅ orders 연관 행도 같이 삭제
```

## 3. JOIN

정규화로 분리된 테이블을 다시 연결해서 조회.

```
[원본 데이터]
users                          orders
user_email       | user_name   order_id | user_email       | product
alice@inu.ac.kr  | Alice              1 | alice@inu.ac.kr  | Keyboard
bob@inu.ac.kr    | Bob                2 | alice@inu.ac.kr  | Mouse
                                        (bob은 주문 없음)
```

### INNER JOIN

양쪽 테이블에 모두 대응되는 행이 있을 때만 결과에 포함.

```sql
SELECT u.user_email, u.user_name, o.order_id, o.product
FROM users u
INNER JOIN orders o ON u.user_email = o.user_email
ORDER BY o.order_id;
```

```
결과: 주문이 없는 Bob은 제외됨
user_email       | user_name | order_id | product
alice@inu.ac.kr  | Alice     | 1        | Keyboard
alice@inu.ac.kr  | Alice     | 2        | Mouse
```

### LEFT JOIN

왼쪽 테이블의 모든 행을 출력. 오른쪽에 없는 값은 NULL로 반환.

```sql
SELECT u.user_email, u.user_name, o.order_id, o.product
FROM users u
LEFT JOIN orders o ON u.user_email = o.user_email
ORDER BY o.order_id;
```

```
결과: 주문이 없는 Bob도 포함, 주문 컬럼은 NULL
user_email       | user_name | order_id | product
alice@inu.ac.kr  | Alice     | 1        | Keyboard
alice@inu.ac.kr  | Alice     | 2        | Mouse
bob@inu.ac.kr    | Bob       | NULL     | NULL
```

### CROSS JOIN (주의)

JOIN 조건 없이 조회하면 두 테이블의 모든 행을 서로 다 조합함.

```sql
-- 아래 두 쿼리는 동일 (의도하지 않은 결과 주의)
SELECT * FROM users u, orders o;
SELECT * FROM users u CROSS JOIN orders o;
```

```
결과: users 2행 × orders 2행 = 4행 (모든 조합)
user_email       | user_name | order_id | product
alice@inu.ac.kr  | Alice     | 1        | Keyboard
alice@inu.ac.kr  | Alice     | 2        | Mouse
bob@inu.ac.kr    | Bob       | 1        | Keyboard
bob@inu.ac.kr    | Bob       | 2        | Mouse
```

## 4. View (가상 테이블)

실제 데이터를 저장하지 않는 가상 테이블. 조회 시점에 원본 테이블을 다시 읽어서 결과를 생성함 (Query Rewrite).

```sql
-- 뷰 생성
CREATE VIEW [뷰이름] AS
SELECT [컬럼들] FROM [테이블] WHERE [조건];

-- 뷰 조회 (일반 테이블처럼 사용)
SELECT * FROM [뷰이름];

-- 뷰 재정의 (있으면 교체, 없으면 생성)
CREATE OR REPLACE VIEW [뷰이름] AS
SELECT [컬럼들] FROM [테이블];
```

> View는 INSERT/UPDATE/DELETE 용도로 쓰지 않음. 조회 전용.

**쓰는 이유 3가지**

1. **복잡한 쿼리 재사용** — 복잡한 JOIN 쿼리를 단순한 명령 하나로 대체
```sql
CREATE VIEW order_summary AS
SELECT u.user_email, u.user_name, o.order_id, o.product, o.qty, o.ordered_at
FROM users u
LEFT JOIN orders o ON u.user_email = o.user_email
ORDER BY o.order_id;

SELECT * FROM order_summary;  -- JOIN 없이 조회 가능
```

2. **DB 구조 변경 시 앱 영향 최소화** — 내부 쿼리가 바뀌어도 앱은 뷰 이름만 바라보면 됨
```sql
-- JOIN 조건이 변경돼도 뷰만 수정하면 됨
CREATE OR REPLACE VIEW order_summary AS
SELECT u.user_email, u.user_name, o.order_id, o.product, o.qty, o.ordered_at
FROM users u
INNER JOIN orders o ON u.user_email = o.user_email
ORDER BY o.order_id;
```

3. **민감한 컬럼 숨기기** — 특정 컬럼(phone 등)을 제외한 뷰를 만들어 접근 제한
```sql
CREATE VIEW customer_view AS
SELECT user_id, email, name, created_at  -- phone 컬럼 제외
FROM customers;
```

## 5. Access Control (권한 관리)

PostgreSQL은 Role 기반으로 사용자와 권한을 관리함.

```sql
-- 사용자 생성 (아래 두 줄은 동일)
CREATE ROLE [사용자명] LOGIN PASSWORD '[비밀번호]';
CREATE USER [사용자명] PASSWORD '[비밀번호]';
```

### GRANT — 3단계 구조

Role 생성 → Role에 권한 부여 → 사용자에게 Role 부여

```sql
-- 1. Role 생성
CREATE ROLE readonly_role;
CREATE ROLE write_role;

-- 2. Role에 권한 부여
-- readonly_role: SELECT만 허용
GRANT USAGE ON SCHEMA public TO readonly_role;
GRANT SELECT ON [테이블명] TO readonly_role;

-- write_role: SELECT + INSERT + UPDATE + DELETE 허용
GRANT USAGE ON SCHEMA public TO write_role;
GRANT USAGE, SELECT ON ALL SEQUENCES IN SCHEMA public TO write_role;
GRANT SELECT, INSERT, UPDATE, DELETE ON ALL TABLES IN SCHEMA public TO write_role;

-- 3. 사용자에게 Role 부여
GRANT readonly_role TO [사용자명];
GRANT write_role TO [사용자명];

-- readonly_role 부여된 사용자
SELECT * FROM users;            -- ✅ 성공
SELECT * FROM user_public_view; -- ❌ 에러 (뷰에 별도 권한 필요)

-- write_role 부여된 사용자
SELECT * FROM user_public_view;                                  -- ✅ 성공 (ALL TABLES에 포함)
INSERT INTO users(email, name) VALUES ('uni@inu.ac.kr', 'Uni'); -- ✅ 성공
```

### REVOKE — 권한 취소

```sql
-- GRANT 반대. TO 대신 FROM 사용
REVOKE readonly_role FROM [사용자명];
REVOKE write_role FROM [사용자명];
```

| 구분 | 명령 |
|------|------|
| 권한 부여 | `GRANT [role] TO [user]` |
| 권한 취소 | `REVOKE [role] FROM [user]` |

> View와 조합하면 강력함 — 원본 테이블 권한은 막고, 민감 컬럼이 제거된 View에만 접근 권한을 부여할 수 있음.

## 핵심 요약

```
제약 조건
  PK      : 행 식별, 중복·NULL 불가, 1개
  UNIQUE  : 중복 불가, NULL 허용, 여러 개
  CHECK   : 비즈니스 로직 강제
  FK      : 테이블 간 관계, 참조 무결성 보장

단일 테이블 문제
  갱신 이상 / 삭제 이상 / 삽입 이상 → 정규화로 해결

정규형 (NF)
  1NF : 모든 컬럼 값이 원자값 (단일 값)
  2NF : 부분 함수 종속 제거 (복합 PK일 때)
  함수 종속 : A → B (A가 결정되면 B도 결정)

JOIN
  INNER JOIN : 양쪽 모두 있는 행만
  LEFT JOIN  : 왼쪽 테이블 전체 + 오른쪽 없으면 NULL
  CROSS JOIN : 모든 조합 (주의)

View
  가상 테이블. 실제 데이터 저장 안 함
  복잡한 쿼리 재사용 / 구조 변경 대응 / 컬럼 숨기기

Access Control
  GRANT  : 권한 부여 (Role → User)
  REVOKE : 권한 취소
```
