# DB | Database Design using the E-R Model

---


## 1. 기본 용어

### 도메인 (Domain)

**도메인**이란 특정 속성(attribute)이 가질 수 있는 **원자적(atomic) 값들의 집합**이다.

```
student_id | name | age
-----------|------|----
2025001    | Kim  | 22
```

| 속성 | 도메인 예시 |
|------|-----------|
| `student_id` | 학번 형식에 맞는 숫자 집합 |
| `name` | 문자열 집합 |
| `age` | 0 ~ 150 사이의 정수 |

>  **핵심**: 실제 저장되는 값은 반드시 해당 속성의 도메인 내에 존재해야 한다. 도메인은 **데이터 타입**과 유사하지만, 값의 범위·의미까지 포함하는 더 넓은 개념이다.

## 2. 데이터베이스 설계 단계

```
요구사항 분석
    ↓
개념적 설계  (E-R 다이어그램 → ERD)
    ↓
논리적 설계  (관계형 모델 → 테이블 스키마)
    ↓
정규화      (이상 현상 제거)
    ↓
물리적 설계  (인덱스, 파티셔닝 등)
```

| 단계 | 도구/결과물 | 설명 |
|------|-----------|------|
| 요구사항 분석 | - | 업무 파악, 데이터 흐름·제약 조건 도출 |
| 개념적 설계 | E-R 다이어그램 (ERD) | DBMS에 독립적인 추상적 설계 |
| 논리적 설계 | 관계형 스키마 (테이블) | DBMS에 종속적인 구체적 설계 |
| 정규화 | 함수 종속 분석 | 이상 현상 제거, 중복 최소화 |
| 물리적 설계 | 인덱스, 파티셔닝 등 | 실제 저장 구조 최적화 |

>  이 강의에서는 **개념적 설계(E-R 모델)**와 **논리적 설계(관계형 모델)**를 중점적으로 다룬다.

## 3. 데이터 모델링 개요

### 데이터 모델링 (Data Modeling)

현실 세계의 정보를 컴퓨터 시스템에 표현하기 위한 **추상화 과정**이다.  
모델링 과정에서는 데이터 구조, 연산 정의, 데이터 형식, 제약 조건, 관계 등을 결정한다.

### 2단계 데이터 모델링

| 구분 | 개념적 데이터 모델링 | 논리적 데이터 모델링 |
|------|-------------------|-------------------|
| 목적 | 현실 세계를 추상적으로 표현 | DBMS에 맞는 구체적 모델 정의 |
| 독립성 | DBMS 독립적 | DBMS 종속적 |
| 대표 도구 | E-R 모델 | 관계형(Relational) 모델 |

### 데이터 모델 (Data Model)

- 현실 세계의 데이터를 **표현하는 방법/도구**
- 사용자와 시스템 사이의 소통 도구

## 4. E-R 모델

### E-R 모델이란?

**붟시-관계 모델(Entity-Relationship Model)**은 Peter Chen이 제안한 **개념적 데이터 모델**이다.

- 개체(Entity)와 개체 간의 관계(Relationship)로 현실 세계를 모델링
- 3가지 핵심 요소: **개체(Entity)**, **속성(Attribute)**, **관계(Relationship)**
- 결과물: **E-R 다이어그램(ERD)**

---

### 4.1 개체 (Entity)

**개체**란 현실 세계에서 독립적으로 존재하며 다른 것과 구별할 수 있는 **유·무형의 사물이나 개념**이다.

| 구분 | 설명 | 예시 |
|------|------|------|
| 유형 개체 | 물리적으로 존재하는 개체 | 학생, 책 |
| 무형 개체 | 논리적/개념적으로 존재하는 개체 | 수업, 주문 |

>  개체는 **속성(Attribute)**으로 설명되며, 개체 인스턴스들은 서로 구별 가능해야 한다.

#### 개체 관련 용어

| 용어 | 설명 | 비고 |
|------|------|------|
| **개체 타입** (Entity Type) | 개체의 구조(이름 + 속성 목록)를 정의 | 틀(클래스 개념) |
| **개체 인스턴스** (Entity Instance) | 특정 개체 타입의 실제 데이터 한 건 | 레코드 1행 |
| **개체 집합** (Entity Set) | 동일 개체 타입의 모든 인스턴스 집합 | 테이블 전체 |

```
개체 타입: 학생(학번, 이름, 나이, 학과)
                ↓
개체 인스턴스: (2025001, Kim, 22, CS)
개체 인스턴스: (2025002, Lee, 21, EE)
                ↓
개체 집합: { (2025001, Kim, ...), (2025002, Lee, ...), ... }
```

#### 강한 개체 vs 약한 개체

| 구분 | 강한 개체 (Strong Entity) | 약한 개체 (Weak Entity) |
|------|--------------------------|------------------------|
| 정의 | 자체 키 속성으로 식별 가능한 개체 | 다른 개체에 의존해야 식별 가능한 개체 |
| 키 | 독립적인 기본키 존재 | 부분 키(partial key) + 소유 개체의 키로 식별 |
| ERD 표기 | 단일 직사각형 | **이중 직사각형** |
| 예시 | `직원(employee_id, 이름)` | `가족(이름, 관계)` — 직원 없이 존재 불가 |

>  약한 개체는 강한 개체와 **전체 참여(total participation)** 관계를 가지며, 두 개체를 연결하는 관계는 **이중 마름모**로 표기한다.

---

### 4.2 속성 (Attribute)

**속성**이란 개체나 관계가 가지는 **특성 또는 성질**이다.  
실제 저장되는 값을 가지며, 관계형 모델에서는 **열(column/field)**에 해당한다.

#### 속성의 분류

**① 단일값 속성 vs 다중값 속성**

| 종류 | 설명 | 예시 | ERD 표기 |
|------|------|------|---------|
| **단일값 속성** (Single-valued) | 하나의 값만 가짐 | 학생의 학번, 주민등록번호 | 타원 1개 |
| **다중값 속성** (Multi-valued) | 여러 값을 가질 수 있음 | 학생의 전화번호, 책의 저자 | **이중 타원** |

>  다중값 속성은 관계형 모델로 변환 시 **별도 테이블**로 분리해야 한다.

**② 단순 속성 vs 복합 속성**

| 종류 | 설명 | 예시 |
|------|------|------|
| **단순 속성** (Simple) | 더 이상 분해되지 않는 원자적 속성 | 학번, ISBN, 나이 |
| **복합 속성** (Composite) | 여러 하위 속성으로 분해 가능한 속성 | 주소(시·구·동·상세주소), 이름(성·이름) |

**③ 저장 속성 vs 유도 속성**

| 종류 | 설명 | 예시 |
|------|------|------|
| **저장 속성** (Stored) | 직접 저장되는 속성 | 생년월일 |
| **유도 속성** (Derived) | 다른 속성으로부터 계산되는 속성 | 나이 (생년월일로부터 계산), 총점 (각 과목 점수의 합) |

>  유도 속성은 ERD에서 **점선 타원**으로 표기한다. 일반적으로 DB에 저장하지 않고 필요할 때 계산한다.

**④ 널 속성 (Null Attribute)**

- 값이 존재하지 않거나 알 수 없는 속성
- `NULL`은 `0`이나 빈 문자열("")과 **다른** 개념
- 예: 배우자 속성이 NULL → 배우자가 없거나 알 수 없음

**⑤ 키 속성 (Key Attribute)**

- 각 개체 인스턴스를 **유일하게 식별**할 수 있는 속성
- ERD에서 **밑줄**로 표기
- 예: 학생 개체의 `학번` 속성

---

### 4.3 관계 (Relationship)

**관계**란 두 개 이상의 개체 사이에 존재하는 **연관성 또는 대응**이다.  
ERD에서 **마름모(Diamond)**로 표기한다.

>  관계에도 속성을 부여할 수 있다. 특히 **M:N 관계**는 관계 자체가 별도의 릴레이션(테이블)으로 변환된다.  
> 예: `수강(수강번호, 수강일자, 학번, ISBN)`

#### 관계의 차수 (Degree)

| 종류 | 참여 개체 수 | 예시 |
|------|------------|------|
| **이진 관계** (Binary) | 2개 | 학생-책(대출), 직원-부서(소속) |
| **단항 관계** (Unary/Recursive) | 1개 | 직원-직원(관리자-부하직원) |
| **삼진 관계** (Ternary) | 3개 | 직원-프로젝트-부서 |

>  삼진 관계는 이진 관계 두 개로 대체가 불가능할 때만 사용한다.  
> 예: `직원-프로젝트-부서` → 이진으로 분리하면 "어느 부서 소속의 직원이 어느 프로젝트에 배정됐는지"를 표현하기 어렵다.

#### 매핑 카디널리티 (Mapping Cardinality)

관계에 참여하는 개체 간의 **수적 대응 관계**를 나타낸다.

| 종류 | 표기 | 설명 | 예시 |
|------|------|------|------|
| **1:1** | 일대일 | A의 인스턴스 1개 ↔ B의 인스턴스 1개 | 직원-사무실 |
| **1:N** | 일대다 | A의 인스턴스 1개 ↔ B의 인스턴스 여러 개 | 부서-직원 |
| **N:M** | 다대다 | A의 인스턴스 여러 개 ↔ B의 인스턴스 여러 개 | 학생-강의 |

```
1:1   A ──── B
1:N   A ──<< B
N:M   A >>──<< B
```

#### 참여 제약 (Participation Constraint)

| 종류 | 설명 | ERD 표기 |
|------|------|---------|
| **전체 참여** (Total Participation) | 모든 개체 인스턴스가 반드시 관계에 참여 | **이중 선** |
| **부분 참여** (Partial Participation) | 일부 개체 인스턴스만 관계에 참여 | 단일 선 |

예: 학생 개체와 대출 관계에서
- 학생은 **부분 참여** → 모든 학생이 반드시 책을 빌릴 필요는 없음
- 책 개체와 대출 관계에서 대출 중인 책은 **전체 참여** → 대출 레코드는 반드시 책이 있어야 함

### 4.4 E-R 다이어그램 표기법

#### Chen 표기법

| 도형 | 의미 |
|------|------|
| 직사각형 | 개체 (Entity) |
| 이중 직사각형 | 약한 개체 (Weak Entity) |
| 마름모 | 관계 (Relationship) |
| 이중 마름모 | 약한 개체의 식별 관계 |
| 타원 | 속성 (Attribute) |
| 이중 타원 | 다중값 속성 |
| 점선 타원 | 유도 속성 |
| 밑줄 속성 | 키 속성 (Key Attribute) |
| 이중 선 | 전체 참여 |
| 단일 선 | 부분 참여 |

#### Crow's Foot 표기법

관계를 **직선**으로 표현하고, 선의 끝 모양으로 카디널리티를 표현한다.

| 기호 | 의미 |
|------|------|
| `||` (이중 수직선) | 정확히 1 (최소 1, 최대 1) |
| `|O` | 0 또는 1 (최소 0, 최대 1) |
| `>|` or `<|` | 1 이상 (최소 1, 최대 N) |
| `>O` or `<O` | 0 이상 (최소 0, 최대 N, zero or many) |

>  다이어그램 상에서 속성은 일반적으로 표기하지 않을 수 있다. (툴마다 다름)

## 5. 관계형 데이터 모델

### 5.1 기본 개념 및 용어

**관계형 데이터 모델(Relational Data Model)**은 현재 가장 널리 사용되는 논리적 데이터 모델이다.  
데이터를 **2차원 테이블(릴레이션)** 형태로 표현한다.

| E-R 모델 용어 | 관계형 모델 용어 | 파일 시스템 용어 |
|-------------|---------------|---------------|
| 개체 타입 | 릴레이션 (Relation/Table) | 파일 (File) |
| 속성 (Attribute) | 열 (Column) | 필드 (Field) |
| 개체 인스턴스 | 튜플 (Tuple) | 레코드 (Record) |

#### 핵심 용어 정리

| 용어 | 설명 |
|------|------|
| **릴레이션 (Relation)** | 동일한 구조의 튜플들의 집합. 2차원 테이블 |
| **속성 (Attribute)** | 릴레이션의 열(column). 데이터의 특성을 나타냄 |
| **튜플 (Tuple)** | 릴레이션의 행(row). 하나의 개체 인스턴스 |
| **도메인 (Domain)** | 속성이 가질 수 있는 원자적 값들의 집합 |
| **널 (NULL)** | 값이 존재하지 않거나 알 수 없음을 나타내는 특수값 (0, ""과 다름) |
| **차수 (Degree)** | 릴레이션의 속성(열) 개수 |
| **카디널리티 (Cardinality)** | 릴레이션의 튜플(행) 개수 |

> 예시: 학생 릴레이션의 **차수(Degree) = 6**, **카디널리티(Cardinality) = 4** (4명의 학생 데이터)

#### 릴레이션 스키마 vs 릴레이션 인스턴스

| 구분 | 설명 | 변경 여부 |
|------|------|---------|
| **릴레이션 스키마** (Relation Schema) | 릴레이션의 구조 정의 (이름 + 속성 목록) | 거의 변경 안 됨 |
| **릴레이션 인스턴스** (Relation Instance) | 특정 시점의 실제 데이터 | 수시로 변경됨 |

```
릴레이션 스키마: 학생(학번, 이름, 학과, 학년, 연락처, 주소)
                        → 구조 정의 (변경 드묾)
릴레이션 인스턴스: 현재 저장된 모든 학생 데이터 행
                        → 실제 데이터 (자주 변경됨)
```

---

### 5.2 릴레이션의 특성

| 특성 | 설명 |
|------|------|
| **튜플의 유일성** | 동일한 튜플이 두 개 이상 존재할 수 없음 |
| **튜플의 무순서** | 튜플 간에 순서가 없음 (행 순서 무의미) |
| **속성의 무순서** | 속성 간에 순서가 없음 (열 순서 무의미) |
| **속성의 원자성** | 모든 속성 값은 원자적(더 이상 분해 불가) 이어야 함 |

>  다중값 속성(예: 전화번호가 여러 개)이 릴레이션에 있으면 **1정규형(1NF)** 위반이다.

---

### 5.3 키 (Key)

**키(Key)**란 릴레이션에서 특정 튜플을 **유일하게 식별**하기 위한 속성 또는 속성들의 집합이다.

#### 키의 특성

| 특성 | 설명 |
|------|------|
| **유일성 (Uniqueness)** | 릴레이션 내의 모든 튜플을 서로 다르게 식별할 수 있어야 함 |
| **최소성 (Minimality)** | 키를 구성하는 속성 중 하나라도 제거하면 유일성이 깨져야 함 |

#### 키의 종류

```
슈퍼키 (Super Key) — 유일성만 만족
    ↓ (+ 최소성)
후보키 (Candidate Key) — 유일성 + 최소성 만족
    ↓ (택 1)
기본키 (Primary Key) — 대표 키로 선정
    ↓ (나머지)
대체키 (Alternate Key) — 기본키로 선정되지 않은 후보키
```

| 키 종류 | 유일성 | 최소성 | 설명 | 예시 (학생 테이블) |
|--------|--------|--------|------|--------------------|
| **슈퍼키** (Super Key) | ✅ | ❌ | 유일성만 만족 | `{학번}`, `{학번, 이름}`, `{이름, 생년월일}` |
| **후보키** (Candidate Key) | ✅ | ✅ | 유일성 + 최소성 | `{학번}`, `{이름, 생년월일}` |
| **기본키** (Primary Key) | ✅ | ✅ | 후보키 중 대표 선정 | `{학번}` |
| **대체키** (Alternate Key) | ✅ | ✅ | 기본키 외 나머지 후보키 | `{이름, 생년월일}` |
| **외래키** (Foreign Key) | ❌ | ❌ | 다른 테이블의 기본키 참조 | 수강 테이블의 `학번` |

>  **복합키(Composite Key)**: 두 개 이상의 속성으로 구성된 키. 후보키, 기본키 모두 복합키가 될 수 있다.

#### 외래키 (Foreign Key) 상세

외래키는 **두 릴레이션 간의 관계를 표현**하는 핵심 메커니즘이다.

```
[수강] 테이블
학번(FK) | 과목코드(FK) | 수강학기 | 성적
---------|------------|---------|----
2025001  | CS101      | 2025-1  | A+
2025001  | CS102      | 2025-1  | B+
2025002  | CS101      | 2025-1  | A

       ↕ FK 참조              ↕ FK 참조
[학생] 테이블           [과목] 테이블
학번(PK) | 이름         과목코드(PK) | 과목명
---------|----         ------------|------
2025001  | Kim         CS101       | 자료구조
2025002  | Lee         CS102       | 알고리즘
```

**외래키 규칙**
1. 참조하는 외래키 값은 반드시 **피참조 테이블의 기본키에 존재**해야 한다.
2. 하나의 릴레이션에 외래키가 **여러 개** 존재할 수 있다.
3. 외래키는 **NULL 값**을 가질 수 있다 (선택적 참조).
4. 외래키와 기본키의 **이름이 달라도** 된다. (도메인이 같으면 됨)

## 6. 무결성 제약 조건

**무결성 제약 조건(Integrity Constraint)**이란 데이터베이스의 **정확성과 일관성을 유지**하기 위한 규칙이다.

| 무결성 종류 | 규칙 |
|-----------|------|
| **개체 무결성** | 기본키는 NULL이 될 수 없다 |
| **참조 무결성** | 외래키는 참조하는 기본키에 존재하는 값이거나 NULL이어야 한다 |

### 개체 무결성 (Entity Integrity)

> **기본키를 구성하는 모든 속성은 NULL 값을 가질 수 없다.**

```sql
-- 위반 예시: 기본키에 NULL 삽입 시도
INSERT INTO 학생(학번, 이름) VALUES (NULL, 'Kim');  -- ❌ 오류 발생
```

**이유**: 기본키가 NULL이면 튜플을 유일하게 식별하는 것이 불가능하기 때문이다.

### 참조 무결성 (Referential Integrity)

> **외래키 값은 참조하는 릴레이션의 기본키에 실제로 존재하는 값이거나 NULL이어야 한다.**

```sql
-- 위반 예시 1: 존재하지 않는 학번 참조
INSERT INTO 수강(학번, 과목코드) VALUES ('9999999', 'CS101');  -- ❌ 9999999 학번 없음

-- 정상: NULL은 허용
INSERT INTO 수강(학번, 과목코드) VALUES (NULL, 'CS101');  -- ✅ (선택 참여인 경우)
```

**참조 무결성 위반 발생 시 처리 방법**

| 처리 방법 | 설명 |
|----------|------|
| `RESTRICT` | 삭제/수정 거부 (오류 발생) |
| `CASCADE` | 참조하는 행도 함께 삭제/수정 |
| `SET NULL` | 외래키를 NULL로 설정 |
| `SET DEFAULT` | 외래키를 기본값으로 설정 |

```sql
-- 예시: 학생이 삭제되면 수강 레코드도 함께 삭제
FOREIGN KEY (학번) REFERENCES 학생(학번) ON DELETE CASCADE
```

### E-R 모델 3요소

| 요소 | ERD 기호 | 설명 |
|------|---------|------|
| 개체 (Entity) | 직사각형 | 독립적으로 존재하는 사물/개념 |
| 속성 (Attribute) | 타원 | 개체/관계의 특성 |
| 관계 (Relationship) | 마름모 | 개체 간 연관성 |

### 키 계층 구조

```
슈퍼키 ⊇ 후보키 ⊇ { 기본키 (1개), 대체키 (0개 이상) }
외래키: 다른 릴레이션의 기본키를 참조하는 속성
```

### 무결성 2원칙

- **개체 무결성**: `PK ≠ NULL`
- **참조 무결성**: `FK 값 ∈ 참조 테이블의 PK 값 집합 ∪ {NULL}`
