# Database System — Introduction

## 1. 데이터 vs 정보

| 용어 | 설명 |
|------|------|
| 데이터 (Data) | 현실에서 관찰·측정한 사실/값. 그 자체로는 의미 제한적 |
| 정보 (Information) | 데이터를 처리해서 의사결정에 유용하게 만든 결과물 |
| 정보 처리 | 데이터 → 정보로 변환하는 과정 |

---

## 2. 데이터베이스 기본 개념

| 용어 | 설명 |
|------|------|
| Database (DB) | 서로 관련된 데이터들의 **조직화된 집합** |
| DBMS | DB를 정의·생성·저장·관리·접근하는 **소프트웨어** |
| Database System | DB + DBMS + Application + Users 전체 시스템 |

### DB의 4가지 정의
- **공유 데이터** : 여러 사용자가 함께 이용
- **통합 데이터** : 중복 최소화
- **저장 데이터** : 컴퓨터가 접근 가능한 매체에 저장
- **운영 데이터** : 조직 운영에 지속적으로 필요한 데이터

### DB의 4가지 특징
- **실시간 접근** : 요청에 즉시 응답
- **계속 변화** : 삽입·삭제·수정으로 최신 상태 유지
- **동시 공유** : 여러 사용자가 같은 데이터 동시 접근
- **내용 기반 참조** : 주소가 아닌 값으로 검색 (`WHERE 재고 >= 1000`)

## 3. 파일 시스템 → DBMS 등장 배경

### 파일 시스템의 문제점

| 문제 | 설명 |
|------|------|
| **데이터 중복성** | 같은 데이터가 여러 파일에 중복 → 일관성·무결성 깨짐 |
| **데이터 종속성** | 파일 구조 바뀌면 응용 프로그램도 전부 수정해야 함 |
| 동시 공유 부족 | 동시 접근 제어 기능 없음 |
| 개발 어려움 | 앱마다 별도 파일 관리 |

### DBMS의 장점

- 데이터 중복 통제
- 데이터 독립성 확보 (앱과 DB 분리)
- 동시 공유 지원
- 보안 향상 (중앙 집중 접근 제어)
- 무결성 유지
- 장애 시 회복 가능
- 개발 비용 절감

### DBMS의 단점

- 별도 구매 비용 발생
- 백업·회복 방법 복잡
- 중앙 집중 → 장애 시 전체 시스템 마비 위험

## 4. DBMS 발전 과정

| 세대 | 종류 | 구조 | 예시 |
|------|------|------|------|
| 1세대 | 네트워크 DBMS | 그래프 형태 | IDS |
| 1세대 | 계층 DBMS | 트리 형태 | IMS |
| 2세대 | **관계 DBMS** | **테이블 형태** | Oracle, MySQL, MariaDB |
| 3세대 | 객체지향 DBMS | 객체 기반 | O2, GemStone |
| 3세대 | 객체관계 DBMS | 관계 + 객체 | Oracle |
| 4세대 | **NoSQL** | 비정형, 분산 | MongoDB, Redis, Cassandra |
| 4세대 | NewSQL | 관계 + NoSQL 확장성 | Spanner, VoltDB |

> 현재 백엔드 개발에서 주력은 **관계 DBMS(MySQL, PostgreSQL)** + 필요시 **NoSQL(Redis, MongoDB)** 조합.

## 5. 3단계 데이터베이스 구조 (ANSI/SPARC)

```
View (External) Level     ← 개별 사용자 관점, 외부 스키마 (여러 개)
        ↕  외부/개념 사상
Logical (Conceptual) Level ← 조직 전체 관점, 개념 스키마 (1개)
        ↕  개념/내부 사상
Physical (Internal) Level  ← 저장 장치 관점, 내부 스키마 (1개)
```

| 단계 | 스키마 | 관점 | 개수 |
|------|--------|------|------|
| 외부 (External) | 외부 스키마 | 개별 사용자 | 여러 개 |
| 개념 (Conceptual) | 개념 스키마 | 조직 전체 | 1개 |
| 내부 (Internal) | 내부 스키마 | 저장 장치 (파일, 인덱스 등) | 1개 |

### 데이터 독립성

| 종류 | 내용 |
|------|------|
| **논리적 독립성** | 개념 스키마 변경 → 외부 스키마 영향 없음 |
| **물리적 독립성** | 내부 스키마 변경 → 개념 스키마 영향 없음 |

> 3단계 구조를 나누는 목적 = **데이터 독립성 실현**

## 6. 데이터 언어 (SQL)

```
SQL = DDL + DML + DCL (+ DQL)
```

| 언어 | 명령어 | 용도 |
|------|--------|------|
| **DDL** (Data Definition) | CREATE, ALTER, DROP | 스키마 정의·수정·삭제 |
| **DML** (Data Manipulation) | SELECT, INSERT, UPDATE, DELETE | 데이터 조작 |
| **DCL** (Data Control) | GRANT, REVOKE 등 | 보안·권한·무결성·회복·동시성 |

> DML은 **비절차적** 언어 — "어떻게"가 아닌 **"무엇을"** 원하는지만 기술 (선언적 언어)

---

## 7. DB 사용자 유형

| 유형 | 역할 | 주로 사용하는 언어 |
|------|------|------------------|
| **DBA** (관리자) | 스키마 정의, 보안, 백업, 성능 감시 | DDL, DCL |
| **최종 사용자** | 데이터 조회·수정 | DML |
| **응용 프로그래머** | 앱 개발, DB 연동 | DML |

---

## 8. 데이터 모델링 흐름

```
현실 세계
    ↓
ER Modeling (개념 설계)
    ↓
ER Diagram
    ↓
Relational Modeling (논리 설계)
    ↓
Relational Schema
    ↓
SQL Implementation (물리 구현)
```

| 모델 | 설명 |
|------|------|
| **ER Model** | 엔티티, 속성, 관계를 시각적으로 표현 |
| **Relational Model** | 테이블(relation) 집합으로 데이터와 관계 표현 |

---

## 핵심 요약

```
파일 시스템 문제 (중복·종속) → DBMS 등장
DBMS = 데이터 중앙 관리 + 독립성 + 동시성 + 보안
3단계 구조 = 외부 / 개념 / 내부 → 데이터 독립성 실현
SQL = DDL(구조) + DML(데이터) + DCL(제어)
모델링 순서 = ER → Relational → SQL
```
