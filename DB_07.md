# 데이터베이스 설계

> 사용자의 요구사항을 고려해 DB 구축

---

## 1. E-R 모델 (Entity-Relationship Model)

> DB 설계 시 구조를 개념적으로 표현하는 방법

### 릴레이션 변환 규칙

> E-R 다이어그램을 실제 테이블로 바꾸는 규칙
>
> 설계 과정 중 이전 단계로 돌아가 설계 내용 변경 가능

---

## 설계 1단계: 요구사항 분석

- 고객의 요구사항을 수집/분석하여 DB 용도 파악
- 사용자가 사용할 범위, 업무 분석

**결과물:** 요구사항 명세서

---

## 설계 2단계: 개념적 설계

- DBMS에 독립적인 개념적 스키마 설계
- 앞선 결과물을 바탕으로 개념적 구조 표현

## 작업 과정

### 1) 개체 추출

> **개체:** 저장할 가치가 있는 주요 데이터
>
> **개체 추출 방법:** 요구사항 명세서 중 업무와 밀접한 명사만 추출
> (중복되거나 관련성이 적은 명사는 배제)

각 개체의 주요 **속성**과 **키 속성** 선별

**예시**

```
요구사항 명세서:
"환자가 의사에게 진료를 받고, 처방전을 발급받는다.
 환자는 이름, 생년월일, 전화번호를 가진다.
 의사는 이름, 전공, 면허번호를 가진다."

① 명사 추출
→ 환자, 의사, 진료, 처방전, 이름, 생년월일, 전화번호, 전공, 면허번호

② 개체 / 속성 분류
→ 개체: 환자, 의사, 처방전       (독립적으로 관리할 대상)
→ 속성: 이름, 생년월일, 전화번호  (개체가 가진 값)
→ 관계: 진료                    (동사 → 다음 단계에서 처리)

③ 키 속성 선별
→ 환자(환자번호, 이름, 생년월일, 전화번호)
→ 의사(면허번호, 이름, 전공)
→ 처방전(처방전번호, 발급일)
```

**STEP 1 결과물**

| 개체명 | 속성 목록 | 키 속성 |
|---|---|---|
| 환자 | 환자번호(PK), 이름, 생년월일, 전화번호(다중값) | 환자번호 |
| 의사 | 면허번호(PK), 이름, 전공 | 면허번호 |
| 처방전 | 처방전번호(PK), 발급일, 약품명 | 처방전번호 |

---

### 2) 개체 간의 관계 설정

> **관계:** 개체 간의 의미 있는 연관성
>
> 앞선 1단계와 달리 여기선 의미 있는 **동사** 추출
>
> **매핑 카디널리티:** 1:1 / 1:N / N:M 관계로 설정
>
> **참여 특성:** 필수적 참여 / 선택적 참여
> (해당 관계가 반드시 존재해야 하는지 여부)

**예시**

```
요구사항 명세서:
"환자가 의사에게 진료를 받고, 처방전을 발급받는다.
 환자는 이름, 생년월일, 전화번호를 가진다.
 의사는 이름, 전공, 면허번호를 가진다."

① 동사 추출
→ 진료를 받는다, 발급받는다

② 관계 도출
→ 진료: 환자 - 의사 사이의 관계
→ 발급: 의사 - 처방전 사이의 관계
→ 수령: 환자 - 처방전 사이의 관계

③ 카디널리티 / 참여 특성 결정
→ 진료: 환자-의사 N:M, 환자 필수 / 의사 선택
→ 발급: 의사-처방전 1:N, 의사 선택 / 처방전 필수
→ 수령: 환자-처방전 1:N, 환자 선택 / 처방전 필수
```

**STEP 2 결과물**

| 관계명 | 개체 A | 개체 B | 카디널리티 | A 참여 | B 참여 |
|---|---|---|---|---|---|
| 진료 | 환자 | 의사 | N:M | 필수 | 선택 |
| 발급 | 의사 | 처방전 | 1:N | 선택 | 필수 |
| 수령 | 환자 | 처방전 | 1:N | 선택 | 필수 |

> 필수 참여 = 반드시 관계에 존재해야 함 | 선택 참여 = 없어도 됨

---

### 3) E-R 다이어그램으로 표현

> STEP 1, 2의 결과물(개체 + 관계)을 시각화한 최종 산출물
>
> DBMS 종류에 관계없이 동일한 표기법 사용

**표기법**

| 기호 | 의미 |
|---|---|
| `[ ]` 사각형 | 개체 (Entity) |
| `< >` 마름모 | 관계 (Relationship) |
| 타원 | 속성 (Attribute) |
| 실선 | 필수적 참여 |
| 점선 | 선택적 참여 |

**예시**

```
[환자]──────<진료>──────[의사]
  |          N:M           |
환자번호               면허번호
이름                   이름
생년월일               전공
전화번호
  |
  | 1:N
  |
[처방전]
처방전번호
발급일
```

> - `환자 - 의사`: N:M 관계 → 나중에 릴레이션 변환 시 중간 테이블 생성
> - `의사 - 처방전`: 1:N 관계 → 처방전 테이블에 FK 추가
> - `환자 - 처방전`: 1:N 관계 → 처방전 테이블에 FK 추가

**결과물:** 개념적 스키마 (E-R 다이어그램)

## 설계 3단계: 논리적 설계

- DBMS에 적합한 논리적 스키마 설계
- 개념적 스키마를 논리적 데이터 모델을 이용해 논리적 구조로 표

## 논리적 설계 변환 규칙

### 1) 모든개체는 릴레이션으로 변환
   
- E-R 다이어그램의 각 개체를 하나의 릴레이션으로 변환
  
- 개체의 이름 -> 릴레이션의 이름
  
- 개체의 속성 -> 릴레이션의 속성
  
- 개체의 키 속성 -> 릴레이션의 기본키
  
- 개체의 속성이 복합속성인 경우에는 복합속성을 구성하고 있는 단순 속성만 릴레이션 속성으로 변환  
   
### 2) n:m 관계는 릴레이션으로 변환

- E-R 다이어그램의 n:m 관계를 하나의 릴레이션으로 변환

- 관계의 이름 -> 릴레이션의 이름

- 관계의 속성 -> 릴레이션의 속성

- 개체를 규칙 1에 따라 변환한 후 릴레이션의 기본키를 관계 릴레이션에 포함시켜 외래키로 지정하고, 외래키들을 조합하여 관계 릴레이션의 기본키로 지정
 
### 3) 1:n 관계는 외래키로 표현
 
### 4) 1:1 관계는 외래키로 표현
 
### 5) 다중 값 속성은 릴레이션으로 변환
 
> 변환규칙을 순서대로 적용하되, 해당되지 않는 규칙은 제외함
