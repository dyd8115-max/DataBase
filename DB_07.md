# 데이터베이스 설계

> 사용자의 요구사항을 고려해 DB 구축

---

## 1. E-R 모델 (Entity-Relationship Model)

> DB 설계 시 구조를 개념적으로 표현하는 방법

### 릴레이션 변환 규칙

> E-R 다이어그램을 실제 테이블로 바꾸는 규칙
>
> 설계 과정 중 이전 단계로 돌아가 설계 내용 변경 가능

---

## 설계 1단계: 요구사항 분석

- 고객의 요구사항을 수집/분석하여 DB 용도 파악
- 사용자가 사용할 범위, 업무 분석

**결과물:** 요구사항 명세서

---

## 설계 2단계: 개념적 설계

- DBMS에 독립적인 개념적 스키마 설계
- 앞선 결과물을 바탕으로 개념적 구조 표현

## 작업 과정

### 1) 개체 추출

> **개체:** 저장할 가치가 있는 주요 데이터
>
> **개체 추출 방법:** 요구사항 명세서 중 업무와 밀접한 명사만 추출
> (중복되거나 관련성이 적은 명사는 배제)

각 개체의 주요 **속성**과 **키 속성** 선별

**예시**

```
요구사항 명세서:
"환자가 의사에게 진료를 받고, 처방전을 발급받는다.
 환자는 이름, 생년월일, 전화번호를 가진다.
 의사는 이름, 전공, 면허번호를 가진다."

① 명사 추출
→ 환자, 의사, 진료, 처방전, 이름, 생년월일, 전화번호, 전공, 면허번호

② 개체 / 속성 분류
→ 개체: 환자, 의사, 처방전       (독립적으로 관리할 대상)
→ 속성: 이름, 생년월일, 전화번호  (개체가 가진 값)
→ 관계: 진료                    (동사 → 다음 단계에서 처리)

③ 키 속성 선별
→ 환자(환자번호, 이름, 생년월일, 전화번호)
→ 의사(면허번호, 이름, 전공)
→ 처방전(처방전번호, 발급일)
```

**STEP 1 결과물**

| 개체명 | 속성 목록 | 키 속성 |
|---|---|---|
| 환자 | 환자번호(PK), 이름, 생년월일, 전화번호(다중값) | 환자번호 |
| 의사 | 면허번호(PK), 이름, 전공 | 면허번호 |
| 처방전 | 처방전번호(PK), 발급일, 약품명 | 처방전번호 |

---

### 2) 개체 간의 관계 설정

> **관계:** 개체 간의 의미 있는 연관성
>
> 앞선 1단계와 달리 여기선 의미 있는 **동사** 추출
>
> **매핑 카디널리티:** 1:1 / 1:N / N:M 관계로 설정
>
> **참여 특성:** 필수적 참여 / 선택적 참여
> (해당 관계가 반드시 존재해야 하는지 여부)

**예시**

```
요구사항 명세서:
"환자가 의사에게 진료를 받고, 처방전을 발급받는다.
 환자는 이름, 생년월일, 전화번호를 가진다.
 의사는 이름, 전공, 면허번호를 가진다."

① 동사 추출
→ 진료를 받는다, 발급받는다

② 관계 도출
→ 진료: 환자 - 의사 사이의 관계
→ 발급: 의사 - 처방전 사이의 관계
→ 수령: 환자 - 처방전 사이의 관계

③ 카디널리티 / 참여 특성 결정
→ 진료: 환자-의사 N:M, 환자 필수 / 의사 선택
→ 발급: 의사-처방전 1:N, 의사 선택 / 처방전 필수
→ 수령: 환자-처방전 1:N, 환자 선택 / 처방전 필수
```

**STEP 2 결과물**

| 관계명 | 개체 A | 개체 B | 카디널리티 | A 참여 | B 참여 |
|---|---|---|---|---|---|
| 진료 | 환자 | 의사 | N:M | 필수 | 선택 |
| 발급 | 의사 | 처방전 | 1:N | 선택 | 필수 |
| 수령 | 환자 | 처방전 | 1:N | 선택 | 필수 |

> 필수 참여 = 반드시 관계에 존재해야 함 | 선택 참여 = 없어도 됨
